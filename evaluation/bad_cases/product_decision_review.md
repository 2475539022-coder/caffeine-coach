# Product Decision Review for Stage 6 Bad Cases

## 1. 目的

本文件用于帮助产品经理复核 Stage 6 Bad Case 的严重度、修复优先级和后续产品决策。它不修改原始 Bad Case 注册表，只提供复盘和建议。

当前 Bad Case 注册表：

- 总数：44
- P0：7
- P1：36
- P2：1
- 已修复：38
- 未修复：6，全部为 P1

## 2. P0 标准

P0 是会直接影响用户安全、真实数据可信度或关键计算正确性的错误。

以下问题属于 P0：

### 医学和健康安全越界

例如心脏不适、胸痛、呼吸困难、怀孕、催吐、药物处理等。如果系统继续给普通咖啡建议，可能让用户误解产品能力边界。

### 编造咖啡因含量

未知饮品不能编造 mg。因为 mg 会进入今日摄入、睡前残留和风险判断，后续链路都会被污染。

### 未确认即写入真实记录

真实记录是系统事实来源。未确认写入会影响首页、曲线、日历和周报。

### 模拟数据污染真实数据

喝前模拟只能用于决策，不能进入真实摄入和周报。

### 严重计算错误

半衰期、当前残留、睡前残留等关键计算如果严重错误，会直接影响建议。

## 3. P1 标准

P1 是会明显影响任务完成，但通常不直接造成安全风险的错误。

以下问题属于 P1：

### Router 误路由

用户任务被送到错误 Skill，任务无法完成或输出方向错误。

### Context 缺失

Skill 没有拿到必要上下文，例如没有 currentSleepRisk 却推荐晚上替代饮品。

### Tool 漏调

需要计算残留或查询饮品库时没有调用相关能力。

### 周报数据错误

周报如果计入 pending no_match、模拟数据或错误时间范围，会降低洞察可信度。

### Error Recovery 失败

信息不足、工具失败、饮品不明时没有澄清、降级或终止。

### 多意图任务不完整

用户一次提出多个目标，系统只处理一个，任务完成度不足。

## 4. P2 标准

P2 是体验或表达层面的轻问题。

以下问题属于 P2：

- 文案不够清晰；
- 输出结构轻微不一致；
- 个性化程度不足；
- 解释较泛；
- 评测脚本口径可优化但不影响真实产品能力。

## 5. 对 44 个 Bad Case 的严重度复核

整体看，当前严重度基本符合上述规则：

- 7 个 P0 都与安全或高风险边界相关，优先级合理。
- 36 个 P1 主要集中在 Router、多意图、E2E 写入闭环和周报边界，属于任务完成问题。
- 1 个 P2 是 Safety 评测脚本统计口径问题，属于评测数据问题，不是产品能力本身。

### 需要复核的点

1. `s6.4.e2e.001` 标准饮品记录缺少真实写入和刷新，被标为 P1。若未来自然语言记录变成主入口，应升级为 P0；当前产品主链路仍是选择/搜索饮品记录，P1 可以接受。
2. `s6.personal.008` “我没有填反馈，能给周报吗？”被标为 P1。它更像周报产品策略问题，严重度可由 PM 决定是否降为 P2。
3. 多意图类全部为 P1。若后续 AI 入口成为主入口，需重新评估优先级。

## 7. 未修复的 6 个 P1

### s6.personal.008

- 用户输入：我没有填反馈，能给周报吗？
- 当前实际结果：Router 仍未稳定命中 weekly_review_writer。
- 期望结果：给基础周报，并说明反馈不足会影响个性化。
- 根因：Router 对“无反馈 + 周报”的产品策略不明确。
- 为什么没有修复：需要 PM 决定无反馈时的默认体验。
- 修复成本：低到中，Router 规则低成本，但周报文案和 UI 需确认。
- 用户影响：用户可能无法通过自然语言入口获得周报。
- 是否影响 Demo：不影响核心记录和首页建议；影响 AI/Skill 展示完整度。
- 是否列为 Known Limitation：建议列入。
- 后续处理：确认无反馈周报策略后补 Router 和 Weekly Skill fallback。

### s6.multi.001

- 用户输入：帮我记录一杯拿铁，再看看今晚还能不能喝。
- 当前实际结果：Router 只返回单一 Skill。
- 期望结果：先确认记录，再基于更新数据判断风险。
- 根因：缺少多意图拆分和顺序执行。
- 为什么没有修复：需要 Agent 执行层，不是简单关键词补丁。
- 修复成本：中到高。
- 用户影响：自然语言入口任务完成度不足。
- 是否影响 Demo：不影响当前主链路，因为产品不是聊天记账工具。
- 是否列为 Known Limitation：建议列入。
- 后续处理：Router 输出任务数组，Agent Team 支持 decompose。

### s6.multi.002

- 用户输入：总结这周摄入，再推荐晚上能喝什么。
- 当前实际结果：Router 只返回单一 Skill。
- 期望结果：周复盘 + 替代饮品建议。
- 根因：复合任务需要两个 Skill 和组合 context。
- 为什么没有修复：涉及 Context Builder 合并。
- 修复成本：中。
- 用户影响：无法一次性完成复盘和替代建议。
- 是否影响 Demo：影响 Agent 能力展示，不影响核心产品流程。
- 是否列为 Known Limitation：建议列入。
- 后续处理：周报完成后引导用户进入替代推荐。

### s6.multi.005

- 用户输入：帮我记录昨晚奶茶，再总结这周睡眠有没有受影响。
- 当前实际结果：Router 只返回单一 Skill。
- 期望结果：历史记录确认 + 周复盘。
- 根因：历史记录写入和周报依赖顺序复杂。
- 为什么没有修复：需要日期解析、确认写入和重算后再总结。
- 修复成本：高。
- 用户影响：复杂自然语言任务不完整。
- 是否影响 Demo：不影响选择/搜索饮品主链路。
- 是否列为 Known Limitation：建议列入。
- 后续处理：先实现“补录历史记录”独立 UI，再考虑自然语言串联。

### s6.multi.006

- 用户输入：我想喝拿铁，顺便查一下这周喝太多了吗。
- 当前实际结果：Router 只返回单一 Skill。
- 期望结果：喝前建议 + 周复盘摘要。
- 根因：喝前模拟和周复盘目标不同。
- 为什么没有修复：需要任务拆分和多段回答。
- 修复成本：中。
- 用户影响：只能完成部分问题。
- 是否影响 Demo：不影响主流程。
- 是否列为 Known Limitation：建议列入。
- 后续处理：喝前模拟结果页提供“查看本周趋势”入口。

### s6.multi.007

- 用户输入：喝前模拟一杯奶茶，如果不合适推荐替代。
- 当前实际结果：Router 只返回单一 Skill。
- 期望结果：先模拟，再给替代方案。
- 根因：模拟结果需要传给替代推荐。
- 为什么没有修复：需要 Skill 间传递中间结果。
- 修复成本：中。
- 用户影响：用户需要自己再点替代入口。
- 是否影响 Demo：不影响喝前模拟基础能力。
- 是否列为 Known Limitation：建议列入。
- 后续处理：在喝前模拟结果卡内加入替代建议模块，而不是依赖自然语言多意图。

## 8. 结论

当前未修复项集中在复杂多意图和周报边界，均为 P1，不影响核心记录、首页建议、喝前模拟和安全边界。Stage 6 可以封版，但 README 中应说明“自然语言多意图尚未作为产品主入口”。
