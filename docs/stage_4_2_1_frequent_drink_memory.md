# Stage 4.2.1：常喝饮品记忆

## 1. 阶段名称

Stage 4.2.1：常喝饮品记忆

## 2. 阶段目标

本阶段目标是让 Caffeine Coach 能根据用户历史记录沉淀“常喝饮品”，并在记录入口提供快捷复用能力，降低用户重复搜索和重复输入成本。

常喝饮品记忆只服务“记录”链路，不改变首页建议、睡前残留计算、状态日历和反馈归属逻辑。

## 3. 新增能力

- 根据历史摄入记录自动生成常喝饮品。
- 用户可以把当前选中饮品手动加入常喝。
- 用户可以从常喝饮品中移除某个饮品。
- 记录一杯弹窗中新增“常喝饮品”快捷区域。
- 点击常喝饮品可按当前时间一键再次记录。
- 删除记录后，自动常喝饮品会根据原始记录重新统计。
- 用户手动固定的常喝饮品不会因为历史记录删除而立即消失。

## 4. 数据结构说明

新增结构：`FrequentDrinkMemory`

字段包括：

- `id`：常喝记忆 id。
- `drinkId`：如果来自饮品库，保留原饮品 id。
- `name`：饮品名称。
- `brand`：品牌或来源。
- `size`：规格。
- `caffeineMg`：咖啡因含量。
- `category`：饮品分类。
- `source`：`auto` 或 `manual`。
- `count`：记录次数。
- `lastUsedAt`：最近一次记录时间。
- `isPinned`：是否用户手动固定。
- `sourceType`：原饮品来源。
- `confidence`：原饮品置信度。
- `isDecaf`：是否低因。

本阶段使用独立 localStorage key：

```text
caffeine-coach-frequent-drinks-v1
```

它不替代原始摄入记录，也不写入 `dailyStatusMemory`。

## 5. 写入规则

### 5.1 自动写入

自动常喝饮品基于原始摄入记录动态计算：

- 近 14 天内同一饮品出现次数大于等于 2 次；
- 或累计记录次数大于等于 3 次；
- 优先使用 `drinkId` 判断同一饮品；
- 如果没有 `drinkId`，使用品牌、名称和咖啡因含量生成稳定 id；
- 同名但咖啡因含量不同的饮品不会被合并。

### 5.2 手动写入

用户可以在记录一杯弹窗中，将当前选中饮品加入常喝。

手动加入后：

- `source = manual`
- `isPinned = true`
- 立即出现在常喝饮品区域
- 不依赖次数阈值

## 6. 更新规则

新增记录后：

- 原有饮品记录正常新增；
- Stage 4.1 状态日历继续重算；
- 常喝饮品统计基于最新 `drinks` 自动更新；
- 已存在的常喝饮品会更新 `count` 和 `lastUsedAt`。

删除记录后：

- 原有饮品记录正常删除；
- Stage 4.1 状态日历继续重算；
- 自动常喝饮品会基于剩余记录重新判断；
- 不再满足阈值的自动常喝饮品会消失；
- 手动固定的常喝饮品不会因为记录减少而自动消失。

## 7. 调用规则

常喝饮品入口位于“记录一杯”弹窗中。

展示规则：

- 最多展示 6 个常喝饮品；
- 优先展示用户手动固定的饮品；
- 其次展示高频和最近使用饮品；
- 展示饮品名称、品牌 / 来源、咖啡因含量、记录次数或固定状态。

点击常喝饮品后：

- 使用该饮品的咖啡因含量；
- 使用当前时间作为摄入时间；
- 走现有新增记录流程；
- 触发首页建议、代谢曲线和状态日历更新。

## 8. 用户控制

用户可以：

- 将当前选中饮品加入常喝；
- 从常喝饮品区域移除某个饮品；
- 移除后，该饮品会进入轻量排除列表，避免短时间内又自动出现；
- 再次手动加入时，会移除排除标记。

## 9. 验收通过项

- 常喝饮品记忆工具已新增。
- 记录一杯弹窗已展示常喝饮品快捷区域。
- 没有历史记录时，常喝区域展示轻量空状态。
- 同一饮品多次记录后，会进入自动常喝候选。
- 同名但咖啡因含量不同的饮品不会被错误合并。
- 点击常喝饮品可一键新增记录。
- 用户可手动加入常喝。
- 用户可从常喝饮品中移除。
- 移除后不会马上重复出现。
- `npm run build` 通过。

## 10. Stage 4.1 回归结果

本阶段未改动 Stage 4.1 的核心机制。

保持不变：

- 新增记录后，状态日历仍会重算。
- 删除记录后，状态日历仍会重算。
- 点击某一天仍能查看当天记录。
- 睡眠反馈与即时不适反馈仍然区分。
- 修改睡眠时间、敏感度、半衰期后，状态仍会重算。
- `dailyStatusMemory` 和 `feedbackMemory` 数据结构未改变。

## 11. 当前边界

本阶段不做：

- `noMatchMemory`
- OCR 记忆
- 复杂饮品库扩充
- 复杂语义匹配
- 复杂推荐算法
- 后端数据库
- 大范围 UI 重构

## 12. 待优化项

- 后续可以在常喝饮品区域增加更细的排序解释。
- 后续可以支持“固定在最前”的显式管理。
- 后续可以在我的页增加常喝饮品记忆管理，但不应影响主记录流程。
- 后续 Stage 4.2.2 再处理 no_match 记忆。

## 13. 阶段结论

Stage 4.2.1 已完成。

Caffeine Coach 现在可以根据用户历史记录沉淀常喝饮品，并在记录入口提供一键复用能力，同时不破坏 Stage 4.1 的状态日历与反馈记忆闭环。
