# Caffeine Coach 产品规则文档 v1.0

## 0. 文档目的

本文件用于约束 Caffeine Coach 后续产品设计、前台页面改动和底层 Agent 能力接入，避免产品在迭代中偏离最初定位。

Caffeine Coach 不是聊天机器人，不是复杂健康报告工具，也不是医疗诊断产品。它是一个轻量咖啡因记录与饮用建议工具，核心帮助用户快速记录摄入，并判断今天是否适合继续摄入咖啡因，尤其关注睡眠影响。

---

## 1. 产品定位

### 1.1 一句话定位

Caffeine Coach 是一个帮助用户记录咖啡因摄入、判断今日饮用风险，并基于作息与反馈给出饮用建议的轻量工具。

### 1.2 核心问题

产品需要帮助用户回答四个问题：

1. 我今天已经摄入了多少咖啡因？
2. 我现在还能不能继续喝？
3. 如果现在喝这一杯，会不会影响今晚睡眠？
4. 我最近对咖啡因是更敏感还是更耐受？

### 1.3 产品不做什么

当前版本明确不做：

1. 不做聊天机器人；
2. 不做医疗诊断；
3. 不做复杂健康报告；
4. 不做无数据支撑的提神变化预测；
5. 不把 Agent Loop / Tool Use / Hooks 等技术机制暴露给普通用户；
6. 不让 AI 自由决定健康结论；
7. 不在主界面展示底层工具调用细节。

---

## 2. 核心用户链路

产品主链路为：

```text
选择 / 搜索 / 识别饮品
↓
确认咖啡因含量和摄入时间
↓
保存记录
↓
首页更新今日状态
↓
系统判断今天还能不能喝
↓
用户可选择喝前模拟或查看原因
```

所有新增功能必须服务以下五类目标之一：
目标
说明
记录
帮助用户更快记录摄入
判断
帮助用户判断今天还能不能喝
建议
给出下一步饮用建议
反馈
收集睡眠、心悸、焦虑等主观反馈
趋势复盘
帮助用户理解长期摄入与睡眠风险变化
如果某个功能不能服务上述任一目标，不应放进用户主界面。

## 3. 页面分工
### 3.1 首页：即时决策
首页只负责回答：

今天还能不能喝？
现在风险高不高？
下一步应该记录一杯、喝前模拟，还是停止摄入？

首页应包含：
1. 今日饮用建议主卡；
2. 记录一杯主入口；
3. 喝前模拟入口；
4. 今日摄入进度；
5. 睡前预计残留；
6. 睡眠风险；
7. 最近记录入口；
8. 轻量“为什么这样建议？”解释入口。
首页不应包含：
1. 多个重复的豆豆状态卡；
2. 重复的今日结论 / 今日建议；
3. 默认展开的大段计算依据；
4. 默认展示的工具调用轨迹；
5. 复杂趋势图；
6. 提神变化曲线；
7. 技术解释文案。

### 3.2 记录页 / 记录弹窗：快速记录
记录模块只负责：

快速找到饮品 → 确认信息 → 保存记录

记录一杯支持三种入口：
入口
说明
搜索输入
用户输入品牌、饮品名、别名或关键词
拍照识别
用户拍饮品包装、小票、菜单截图，系统识别候选
手动输入
饮品库查不到时，用户手动输入名称和咖啡因含量
记录页不应默认展示完整饮品库列表。
系统应根据用户输入推荐候选，而不是一开始铺开所有数据。

### 3.3 喝前模拟：摄入前决策
喝前模拟负责回答：

如果我现在喝这一杯，会不会影响睡眠？

喝前模拟展示：
1. 选中饮品；
2. 该饮品咖啡因含量；
3. 喝完后今日累计；
4. 睡前预计残留；
5. 风险等级；
6. 行动建议。
喝前模拟不应展示：
1. 大段技术解释；
2. Agent Loop 文案；
3. 工具调用函数名；
4. 复杂长期趋势图。

### 3.4 洞察页：长期趋势复盘
洞察页只负责长期分析，不承担首页即时决策。
洞察页应保留三类信息：
模块
回答的问题
近 7 天摄入趋势
最近喝得多不多
睡前残留趋势
最近是否经常睡前残留偏高
耐受 / 敏感度趋势
用户最近更敏感还是更耐受
可选保留：
1. 晚间摄入次数；
2. 超过建议量天数；
3. 睡眠 / 心悸 / 焦虑反馈统计；
4. 常喝饮品分布。
洞察页不应默认展示：
1. 无数据支撑的提神变化曲线；
2. 复杂健康分析；
3. 医疗化结论；
4. 与咖啡因记录和睡眠风险无关的内容。

### 3.5 我的页：个人设置
我的页负责用户基础设置：
1. 睡眠时间；
2. 起床时间；
3. 敏感度；
4. 今日推荐摄入目标；
5. 常喝饮品；
6. 数据管理；
7. 反馈偏好。

## 4. 首页信息架构规则
### 4.1 首页只允许一个核心状态中心
首页第一屏只保留一张核心主卡：

今日饮用建议

该卡回答：

今天还能不能喝？

核心主卡包含：
1. 咖啡豆状态；
2. 睡眠风险标签；
3. 核心结论；
4. 最多 3 个关键数据；
5. 记录一杯按钮；
6. 喝前模拟按钮；
7. 为什么这样建议入口。
### 4.2 首页关键数据最多展示 3 个
首页主卡默认最多展示：
指标
说明
今日已摄入
用户今天已记录的咖啡因总量
睡前预计残留
按当前摄入和作息估算的睡前体内剩余
今日目标 / 还可摄入
当前个性化推荐目标及剩余额度
不要在首页多个位置重复展示这些指标。
### 4.3 进度卡规则
如果保留“今日摄入进度卡”，它只展示：
1. 已摄入 / 推荐量；
2. 还可摄入。
进度卡不再重复写：
1. 今天还能不能喝；
2. 睡眠风险高；
3. 建议停止摄入；
4. 今日结论。
### 4.4 解释入口规则
“为什么这样建议？”默认收起。
展开后展示：
1. 原因；
2. 依据数据；
3. 风险标签；
4. 查看计算依据。
工具调用轨迹只能放在更深层折叠：

开发调试：查看工具调用轨迹

并且默认收起。

## 5. 记录一杯交互规则
### 5.1 记录流程
标准记录流程：

打开记录一杯
↓
搜索 / 拍照 / 手动输入
↓
系统返回候选
↓
用户选择饮品
↓
确认咖啡因含量和摄入时间
↓
保存记录
↓
首页状态更新

### 5.2 搜索记录规则
用户输入品牌、饮品名、别名或关键词后，系统返回可能候选。
候选卡片展示：
1. 饮品名称；
2. 品牌；
3. 咖啡因含量；
4. 来源：品牌库 / 食品库 / 估算 / 用户自定义；
5. 置信度：高 / 中 / 低；
6. 杯型或容量信息。
不允许默认展示完整饮品库列表。
### 5.3 拍照识别规则
拍照识别支持：
1. 饮品包装；
2. 菜单；
3. 小票；
4. 截图。
识别后只返回候选，不直接写入记录。
若识别结果不确定，必须让用户确认。
### 5.4 手动输入规则
当饮品库无匹配时，用户可以手动输入：
1. 饮品名称；
2. 咖啡因含量；
3. 摄入时间；
4. 备注，可选。
手动输入内容可进入后续 no_match 历史或常喝饮品记忆。
### 5.5 确认记录弹窗规则
确认弹窗默认只展示：
1. 饮品名称；
2. 咖啡因含量；
3. 摄入时间；
4. 保存记录按钮。
以下内容放入“高级设置”，默认收起：
1. 手动修改咖啡因含量；
2. 备注；
3. 置信度详细说明；
4. 饮品类型；
5. 来源说明；
6. 杯型补充。
保存按钮必须固定在底部，用户不应滑到最底部才能保存。

## 6. 饮品匹配规则
### 6.1 匹配状态
饮品搜索返回三种状态：
状态
含义
产品处理
exact_match
明确匹配
可直接进入确认记录
fuzzy_match
模糊匹配
需要用户确认候选
no_match
无可信匹配
不允许编造，要求手动输入或补充信息
### 6.2 no_match 规则
当饮品无可信匹配时：
1. 不返回伪候选；
2. 不编造咖啡因含量；
3. 不继续计算风险；
4. 提示用户补充品牌、杯型或手动输入咖啡因 mg；
5. 可记录到 no_match 历史中，后续用于补库。
### 6.3 置信度展示规则
置信度只作为辅助信息，不应抢占主流程。
置信度
展示方式
高置信
轻量标签
中置信
轻量标签，可提示确认
低置信
必须允许手动修改咖啡因含量

## 7. 今日推荐量规则
### 7.1 推荐量定义
今日推荐摄入量不是固定值，应由基础建议量与个体校准共同决定。
推荐量用于回答：

今天建议摄入上限是多少？

### 7.2 推荐量组成
推荐量由以下因素决定：

今日推荐量 = 基础建议量 × 个体校准因子 × 时间风险因子 × 反馈修正因子

实际实现中不一定要暴露公式给用户，但产品解释必须说明推荐量来自：
1. 用户设置；
2. 睡眠时间；
3. 咖啡因敏感度；
4. 最近摄入记录；
5. 最近反馈；
6. 睡前残留风险。
### 7.3 基础建议量
基础建议量可来自用户设置，若用户未设置，则使用系统默认值。
基础建议量不直接等同于医学建议，仅作为产品内风险管理目标。
### 7.4 个体校准因子
个体校准因子根据用户敏感度和近期反馈调整：
用户状态
调整方向
高敏感
下调推荐量
中敏感
保持基础推荐量
低敏感 / 耐受较高
可轻微上调
最近频繁反馈睡不好
下调
最近频繁反馈心悸 / 焦虑
下调
长期无不适反馈
保持或轻微上调
### 7.5 时间风险因子
时间越接近睡眠时间，可摄入量越保守。
例如：
距离睡眠时间
策略
> 8 小时
正常
4–8 小时
轻度保守
2–4 小时
明显保守
< 2 小时
建议停止摄入
### 7.6 用户可见解释
用户不需要看到复杂公式。
应展示为：

今日推荐量根据你的睡眠时间、敏感度和近期反馈估算。
你最近睡前残留偏高，因此今日目标较保守。


## 8. 睡眠风险规则
### 8.1 风险等级
睡眠风险分为：
风险
含义
低风险
睡前残留较低，可正常摄入或少量摄入
中风险
睡前残留可能偏高，建议控制分量
高风险
睡前残留明显偏高，建议停止摄入
### 8.2 风险判断依据
风险主要依据：
1. 今日已摄入；
2. 当前体内剩余；
3. 睡前预计残留；
4. 距离睡眠时间；
5. 用户敏感度；
6. 最近反馈。
### 8.3 健康表达边界
文案必须克制：
可以说：

可能影响睡眠
风险偏高
建议谨慎
建议今晚先暂停摄入

不可以说：

一定会失眠
你已经咖啡因中毒
你有健康问题
你必须就医

如用户反馈严重不适，例如胸痛、严重心悸、呼吸困难，应建议停止继续摄入并咨询专业人士。

## 9. 咖啡豆状态规则
### 9.1 咖啡豆状态定位
咖啡豆状态是前台拟人化表达层，用于降低产品冰冷感。
它不直接做健康判断。
咖啡豆状态由规则决定，AI 只负责文案润色。
### 9.2 状态数量
当前版本建议保留 5 种状态：
状态
含义
清醒豆
当前状态稳定，可以正常或少量摄入
纠结豆
还能喝，但建议控制分量
焦虑豆
负荷偏高，建议停止摄入
恢复豆
已进入恢复期，建议不再摄入
低因豆
可选择低因、半杯、小杯方案
### 9.3 状态触发规则
清醒豆
触发条件：
	•	今日摄入低于推荐量 50%；
	•	睡前预计残留低；
	•	睡眠风险低；
	•	近期无明显不适反馈。
展示建议：

今天状态比较稳定，可以正常安排咖啡因摄入。


纠结豆
触发条件：
	•	今日摄入接近推荐量；
	•	或睡前预计残留中等；
	•	或睡眠风险中；
	•	还可以少量摄入，但不建议完整一杯高咖啡因饮品。
展示建议：

还能喝，但建议控制分量，优先半杯或小杯。


焦虑豆
触发条件：
	•	今日摄入已超过推荐量；
	•	或睡前预计残留高；
	•	或睡眠风险高；
	•	或近期存在睡不好、心悸、焦虑等反馈。
展示建议：

今天咖啡因负荷偏高，建议先停止摄入。


恢复豆
触发条件：
	•	今日已达到或超过推荐量；
	•	距离最后一次摄入已有一段时间；
	•	当前重点是降低残留，而不是继续摄入；
	•	适合提醒用户补水、休息、避免继续刺激。
展示建议：

现在更适合进入恢复状态，今晚先不继续摄入。


低因豆
触发条件：
	•	用户仍有提神需求；
	•	但完整一杯咖啡因饮品风险偏高；
	•	系统推荐低因、半杯、小杯或无因替代。
展示建议：

如果确实需要提神，建议选择低因、半杯或小杯。

### 9.4 状态优先级
当多个状态同时满足时，按风险优先级判断：

焦虑豆 > 恢复豆 > 纠结豆 > 低因豆 > 清醒豆

### 9.5 AI 的作用边界
AI 可以参与：
1. 改写状态文案；
2. 生成更自然的解释；
3. 把规则结果组织成用户能理解的话。
AI 不应参与：
1. 自由决定状态类型；
2. 编造风险原因；
3. 替代规则判断；
4. 输出医疗诊断。

## 10. 洞察页规则
### 10.1 洞察页目标
洞察页回答：

我最近的咖啡因摄入和睡眠风险有什么规律？

不回答：

我是不是身体有问题？
我是不是咖啡因成瘾？
我未来一定会怎样？

### 10.2 洞察页核心模块
近 7 天摄入趋势
展示：
1. 每日摄入量；
2. 超过推荐量天数；
3. 晚间摄入次数。
睡前残留趋势
展示：
1. 每日睡前预计残留；
2. 高风险天数；
3. 与睡眠时间的关系。
耐受 / 敏感度趋势
展示：
1. 当前敏感度等级；
2. 当前推荐量为何调整；
3. 最近反馈如何影响推荐量。
反馈关联
展示：
1. 睡眠差次数；
2. 心悸次数；
3. 焦虑次数；
4. 与晚间摄入的简单关联。
### 10.3 洞察页不应展示
1. 无反馈数据支撑的提神变化曲线；
2. 医疗化分析；
3. 复杂诊断结论；
4. 和咖啡因摄入无关的健康状态判断。

### 11.3 删除记录规则
用户点击删除记录时：
1. 不直接删除；
2. 弹出确认；
3. 说明删除会影响今日摄入、残留计算和趋势统计；

5. 点击确认删除才删除；
6. 删除后刷新首页与洞察数据。

## 12. Hooks 规则

Hooks 用于：

数据变化后，统一刷新首页建议、今日摄入、残留和洞察统计。

## 13. Agent Loop / Tool Use 使用边界
### 13.1 Agent Loop 的作用
Agent Loop 是底层建议调度机制，不是前台聊天入口。
它用于：
1. 根据当前场景判断要调用哪些工具；
2. 汇总今日记录、用户画像、残留计算和推荐摘要；
3. 生成首页建议、喝前模拟解释和依据说明。
### 13.2 Tool Use 的作用
Tool Use 把业务能力封装成工具：
1. 读取今日记录；
2. 读取用户画像；
3. 查询饮品库；
4. 计算当前残留；
5. 模拟喝前风险；
6. 生成推荐摘要。
### 13.3 前台展示原则
普通用户默认不看到：
1. Agent Loop；
2. Tool Use；
3. TOOL_HANDLERS；
4. usedTools；
5. 函数名；
6. 调试轨迹。
用户看到的是：
1. 结论；
2. 原因；
3. 依据数据；
4. 建议行动。

## 16. 改动准入规则
后续任何页面或功能改动前，必须回答：
1. 它服务记录、判断、建议、反馈、趋势复盘中的哪一个？
2. 它是否会让首页更复杂？
3. 它是否依赖真实数据？
4. 它是否只是为了展示技术机制？
5. 它是否会让用户误以为产品是聊天机器人或医疗诊断工具？
6. 它是否会重复展示已有信息？
7. 它是否影响主链路效率？
如果答案不明确，不应进入用户主界面。

## 17. 当前最高优先级
当前最优先的工作不是继续加新功能，而是：
1. 收敛首页重复信息；
2. 重构记录一杯为搜索 / 拍照 / 手动输入三入口；
3. 恢复并明确耐受 / 敏感度趋势；
4. 明确咖啡豆状态规则；
5. 补充删除记录二次确认；
6. 用轻量 Hooks 保证记录、删除、反馈后状态刷新。
