# Stage 6.6 Evaluation Transparency Review

## 1. 阶段名称

Stage 6.6：评测透明化与产品决策复盘

## 2. 阶段目标

本阶段不继续优化代码，而是把 Stage 6 的评测集设计、初始结果、Bad Case 归因、修复动作和修复前后变化透明化，让产品经理能够理解并参与关键决策。

新增文档包括：

- `evaluation/datasets/dataset_design_rationale.md`
- `evaluation/results/stage_6_before_after_examples.md`
- `evaluation/bad_cases/product_decision_review.md`
- `docs/stage_6_6_evaluation_transparency_review.md`

## 3. Stage 6 评测方法论

可用于面试或作品集讲述的方法如下：

1. 根据用户任务和风险设计评测集。
2. 定义预期 Skill、工具、上下文和通过标准。
3. 分模块跑 Router、Context、Tool、Safety。
4. 再跑端到端任务。
5. 将失败项归因为 Router / Context / Tool / Skill / Safety / Recovery。
6. 根据用户影响划分 P0 / P1 / P2。
7. 比较多种修复方案。
8. 做最小修复。
9. 原用例复测。
10. 全量回归。

## 4. Codex 执行了哪些工作

Codex 在 Stage 6 中主要负责：

- 建立 EvaluationTask 协议；
- 生成设计型离线评测集；
- 编写模块级和端到端评测 runner；
- 输出初始结果；
- 汇总 Bad Case；
- 对 P0 / P1 做最小代码修复；
- 生成复测结果和 Stage 6 总结；
- 本阶段补充透明化复盘文档。

## 5. 产品经理应该负责哪些判断

产品经理需要参与：

- 测试集是否覆盖真实高频任务；
- 每个 Bad Case 是否真的影响用户任务；
- 严重度是否合理；
- 修复方案是否符合产品定位；
- 是否应该追问用户，而不是自动推断；
- 多意图是否进入当前 Demo 范围；
- 哪些 Known Limitation 需要在 README 中公开说明。

## 6. 当前仍需要 PM 人工复核的决策

重点复核清单见 `evaluation/bad_cases/product_decision_review.md`。主要问题包括：

- 无反馈时是否直接生成基础周报；
- 自然语言多意图是否进入产品主链路；
- 多意图任务应该由 Agent 拆分，还是由 UI 入口承接；
- 高风险安全问题是否需要固定强提示模板；
- 泛称饮品是否由前端候选卡承接澄清；
- Safety 评测是否保留“全量安全扫描”作为独立指标。

## 7. 指标提升拆解

### 7.1 Router Accuracy：48.3% → 90%

#### 系统能力真实提升

- 补充了喝前模拟、明确剂量、敏感度、周报、替代饮品、高风险安全表达等路由规则。
- 典型修复包括“帮我模拟一下喝奶茶”“每天 800mg 可以吗？”“我现在很焦虑，但想靠咖啡提神”。

#### 评测脚本或统计口径修正

Router 指标主要来自规则补强，不是统计口径修正。

#### 测试集覆盖变化

测试集没有被删除或简化；仍保留多意图困难用例。

#### 是否可能存在指标虚高

存在一定可能。当前 Router 仍以关键词为主，面对真实用户表达的泛化能力有限。

#### 当前指标适用范围

适用于 Stage 6.1 设计型离线测试集，不代表真实线上准确率。

### 7.2 E2E Task Success Rate：33.3% → 83.3%

#### 系统能力真实提升

- Router 能把更多任务送到正确 Skill。
- E2E runner 能识别模拟、工具失败、数据不足等恢复动作。

#### 评测脚本或统计口径修正

部分提升来自 E2E runner 对已定义恢复动作的识别补充，而不是产品运行时新增写入能力。

#### 测试集覆盖变化

测试集保持不变，多意图和真实写入闭环仍保留为失败项。

#### 是否可能存在指标虚高

有。离线 E2E 评测不执行真实用户确认写入，也不真实刷新首页、曲线和日历。

#### 当前指标适用范围

适用于离线 Agent Team 链路，不等同于完整前端交互验收。

### 7.3 Recovery Success Rate：54.5% → 100%

#### 系统能力真实提升

- 明确了 clarify、fallback、degrade、stop、decompose 的触发场景。
- 信息不足、工具失败和安全问题都能进入对应恢复动作。

#### 评测脚本或统计口径修正

提升中包含 runner 对恢复动作的显式识别补充。

#### 测试集覆盖变化

未删除工具失败、多意图、安全问题等困难用例。

#### 是否可能存在指标虚高

存在。当前恢复动作是离线判定，不代表 UI 已完整承接所有恢复交互。

#### 当前指标适用范围

适用于恢复策略是否被识别，不代表用户交互闭环全部完成。

### 7.4 Safety Pass Rate：31.7% → 100%

#### 系统能力真实提升

- Safety Reviewer 增加了心脏不适、胸痛、呼吸困难、心慌、手抖、胃不舒服、怀孕、催吐、高剂量、青少年能量饮料等边界识别。
- 高风险场景会降级为停止继续摄入和必要时咨询专业人士。

#### 评测脚本或统计口径修正

这是最需要拆开的指标。

原始 Safety runner 把全量任务都纳入 Safety Pass Rate，导致普通记录任务因为“没有触发安全升级”被误计为失败。Stage 6.5 修正为只评测 `category = safety` 的用例。

因此，Safety Pass Rate 的提升来自两部分：

1. 真实安全规则补强；
2. 统计口径修正。

不得把 31.7% → 100% 全部解释为系统能力提升。

#### 测试集覆盖变化

安全测试集没有删减，仍包含药物、怀孕、过敏、心脏不适、高剂量、保证不失眠等问题。

#### 是否可能存在指标虚高

有。当前是设计型安全集，不能代表所有真实健康问题表达。

#### 当前指标适用范围

适用于 Stage 6.1 safety category，不适用于医疗安全全面认证。
