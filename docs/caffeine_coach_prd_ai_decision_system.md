# Caffeine Coach 咖啡因 AI 饮用决策助手 PRD v1.0

## 文档使用说明

这是一份面向作品集、面试讲述和后续产品迭代的阶段性 PRD。它不是一个普通咖啡因记录 App 的功能清单，而是围绕“用户准备喝一杯时，如何判断现在还能不能喝、喝多少更合适、为什么这样建议”来定义一个可实现、可验证、可评测的 AI 产品方案。

读这份文档时，可以重点看三件事：

1. Caffeine Coach 为什么不是简单的咖啡因计算器；
2. 真实 AI 只介入哪一个高摩擦节点；
3. 第一版如何在不改变规则结论的前提下，通过结构化证据、服务端 LLM、校验与 fallback，生成可解释的个性化喝前建议。

## 核心产品定位

Caffeine Coach 不是一个“AI 聊天教练”，也不是复杂健康报告工具。第一版聚焦一条更有产品厚度的链路：

```text
准备喝一杯
-> 选择 / 搜索 / 确认饮品
-> 规则计算本次摄入后的残留与风险
-> 输出整杯 / 半杯 / 低因 / 不喝建议
-> 用户查看为什么这样建议
-> AI 基于历史证据生成个性化解释
-> 用户确认行动
-> 状态、反馈和记忆沉淀
-> 下一次建议更贴近用户
```

第一版要证明的核心不是“AI 能不能帮用户聊天”，而是：

> AI 能否在确定性规则已经给出喝前决策后，把用户的近期饮用与反馈证据转化为克制、可信、可追溯的个性化解释，帮助用户理解并执行当前建议。

# 一、项目定义与业务场景

这一部分回答项目为什么成立：目标用户是谁，在什么时刻遇到什么任务，哪里是高摩擦节点，AI 为什么有必要介入，以及第一版 MVP 应该证明什么。

## 1.1 项目概述

### 一句话定义

Caffeine Coach 面向关注睡眠影响、对咖啡因摄入较敏感或希望减少摄入的用户，在用户准备喝咖啡、奶茶、茶饮或能量饮料前，基于今日摄入、饮品咖啡因含量、作息、敏感度和近期反馈，帮助用户判断“现在还能不能喝、喝多少更合适”，并给出可执行饮用建议。

### 项目要解决的问题

很多咖啡因工具解决的是“记录喝了多少 mg”，但用户真正反复卡住的是：

- 我不知道这杯饮品大概有多少咖啡因；
- 我今天已经喝过几杯了，现在还能不能继续喝；
- 如果现在喝，会不会影响今晚睡眠；
- 为什么系统今天建议半杯、低因或不喝；
- 我感觉自己有时心慌、睡不好，但不知道这些反馈如何影响建议。

只给用户一个咖啡因数字或代谢曲线，不足以帮助用户完成“当前这杯怎么喝”的决策。

### 第一版核心任务

第一版不追求完整健康管理，也不追求医学级判断。它只验证一条主任务：

> 用户准备喝一杯时，系统能否先用规则稳定算出风险和行动建议，再用 AI 解释“为什么本次建议适合你”。

## 1.2 目标用户与触发场景

### 目标用户

| 用户类型 | 典型状态 | 核心需求 | 失败成本 |
| --- | --- | --- | --- |
| 睡眠敏感用户 | 下午或晚上喝咖啡后容易睡不好 | 判断最晚饮用时间和睡前残留风险 | 晚上睡眠受影响，第二天状态变差 |
| 咖啡/奶茶高频用户 | 经常一天多杯，但不清楚累计量 | 快速记录并知道今天还能不能喝 | 超过个人目标，摄入失控 |
| 减少咖啡因用户 | 想少喝，但不想完全戒掉 | 获得半杯、低因、小杯等替代动作 | 建议过于极端导致放弃记录 |
| 对咖啡因不适用户 | 偶尔心慌、焦虑、手抖或胃不舒服 | 理解主观反馈如何影响建议 | 系统误把预测风险说成用户已不适 |
| 数据记录型用户 | 愿意记录饮品和反馈 | 看近 7 天/14 天趋势与状态日历 | 记录无法回流，用户觉得系统没有记忆 |

### 核心触发场景

| 触发时刻 | 用户输入或动作 | 系统应该做什么 | 输出结果 |
| --- | --- | --- | --- |
| 喝前 | 选择一杯拿铁、奶茶、美式等 | 模拟喝完后的累计摄入、睡前残留和风险 | 整杯 / 半杯 / 低因 / 不喝建议 |
| 喝后 | 点击记录一杯，搜索或选择饮品 | 匹配饮品库或自定义饮品，确认咖啡因和时间 | 写入记录并刷新今日状态 |
| 查看首页 | 打开 App | 汇总今日摄入、剩余额度、睡前残留和风险 | 今日饮用建议 |
| 想理解建议 | 点击“为什么这样建议？” | 展示规则依据；证据足够时调用 AI 个性化解释 | 规则说明 + 个性化历史观察 |
| 复盘趋势 | 进入洞察页 / 状态日历 | 查看近 7 天趋势和近 14 天每日状态 | 趋势图、日历详情、反馈补录 |
| 饮品库找不到 | 搜索未知饮品 | 不编造咖啡因，询问是否创建自定义饮品 | no_match / custom drink 兜底 |

## 1.3 业务痛点与失败成本

### 痛点 1：用户不知道一杯饮品有多少咖啡因

用户通常记得“喝了生椰拿铁、伯牙绝弦、便利店咖啡”，但不知道具体 mg。如果产品一开始要求手动输入咖啡因含量，记录门槛会很高。

第一版解决方式：

- 饮品库搜索优先；
- 常喝饮品快捷复用；
- 自定义饮品沉淀；
- no_match 不编造，用户确认后再保存。

### 痛点 2：代谢曲线和睡眠风险关系不直观

用户可能看得懂“睡前预计残留 68mg”，但不一定知道这意味着应该喝半杯、低因还是停止摄入。

第一版解决方式：

- 规则层计算当前剩余和睡前预计残留；
- 喝前模拟直接输出行动建议；
- 首页只保留一个核心状态表达；
- 复杂原因默认收起。

### 痛点 3：不同用户对咖啡因反应差异明显

同样一杯拿铁，有的人没感觉，有的人会心慌或睡不好。固定通用阈值会让建议显得不贴近用户。

第一版解决方式：

- 用户设置作息、敏感度、提醒严格度和半衰期；
- 主观反馈区分睡眠反馈与即时不适反馈；
- 状态日历沉淀每日风险；
- AI 解释只在证据达到门槛时引用历史观察。

### 痛点 4：“为什么不建议喝”容易变成黑箱

如果系统只说“风险高，不建议喝”，用户可能觉得生硬；如果 AI 自由解释，又可能编造数字或制造焦虑。

第一版解决方式：

- 规则层先给确定结论；
- LLM 只解释，不改结论；
- 所有数字来自 allowedFacts；
- 服务端校验拦截内部枚举、非法数字、因果化、医学化和绝对化表达；
- 失败时回退规则解释。

## 1.4 AI 价值与必要性

### 非 AI 方案能解决什么

- 饮品库可以查咖啡因含量；
- 半衰期公式可以计算当前残留；
- 规则阈值可以判断低 / 中 / 高风险；
- 前端卡片可以展示今日建议、进度和趋势；
- localStorage 可以沉淀记录、反馈和常喝饮品。

### 非 AI 方案解决不了什么

- 难以从多日记录和反馈里选择“与本次决策最相关”的历史观察；
- 难以用自然语言解释“为什么今天比平时更保守或更宽松”；
- 难以表达证据限制，避免把共现说成因果；
- 难以在正向决策时处理负向历史证据的对照关系；
- 难以把规则依据、历史观察和行动建议组织成用户愿意读的短解释。

### AI 只负责什么

AI 第一版只负责“AI 个性化喝前解释”：

- 从规则层提供的脱敏结构化历史证据中，选择与本次喝前决策最相关的一条观察；
- 解释为什么当前规则建议适合这次场景；
- 说明数据限制和不确定性；
- 生成一条与规则结论一致的行动表达。

### AI 不负责什么

AI 不得：

- 重新计算咖啡因残留、睡前残留、风险等级或今日累计；
- 改变或放宽规则结论；
- 编造饮品、记录、反馈或咖啡因含量；
- 把历史共现写成因果；
- 修改用户敏感度、推荐量、画像或真实记录；
- 做医学诊断；
- 生成聊天式长回答。

## 1.5 MVP 功能边界

### 第一版 MVP

第一版 MVP 由两个层次组成：

1. 规则型产品 baseline：记录、模拟、计算、趋势、反馈、记忆；
2. AI 增强层：喝前模拟页的“为什么这样建议？”个性化解释。

第一版只在以下条件同时满足时调用 LLM：

- 用户主动展开“为什么这样建议？”；
- 最近 14 天数据达到最低证据门槛；
- 当前 dataVersionHash 没有可复用缓存；
- 当日真实模型调用未超过限制；
- 请求通过服务端输入校验。

### 自进化接口

这里的“自进化”不是模型自动训练，而是产品状态回流：

- 用户记录饮品后，影响今日摄入、睡前残留、状态日历和常喝饮品；
- 用户保存反馈后，影响敏感度解释和后续建议；
- 用户创建自定义饮品后，后续搜索和记录可以复用；
- 用户对解释“不符合”可进入 Bad Case，不自动训练模型。

### 主动服务接口

当前版本不做系统推送。主动服务只体现在界面状态：

- 首页默认给今日饮用建议；
- 喝前模拟主动提示整杯、半杯、低因或不喝；
- 状态日历提示某天风险和反馈缺口；
- 洞察页提示近 7 天趋势。

# 二、产品方案与技术实现

这一部分回答 MVP 从触发、输入、规则计算、AI 解释、展示、反馈到状态沉淀如何完整跑起来。

## 2.1 轻量竞品 / 替代方案调研

第一版不追求复刻完整健康平台，而是避开“大而全”路线，聚焦一个小闭环。

| 替代方案 | 可借鉴 | 不足 | Caffeine Coach 切入点 |
| --- | --- | --- | --- |
| 普通咖啡因计算器 | 咖啡因半衰期、残留估算 | 用户仍需自己理解数值 | 把数值转成可执行饮用建议 |
| 饮食 / 健康记录 App | 日志、趋势、目标设置 | 咖啡因场景不够细，喝前决策弱 | 聚焦“现在这杯怎么喝” |
| 睡眠 App | 睡眠时间、睡眠质量反馈 | 不追踪咖啡因摄入细节 | 连接摄入时间、残留和睡眠风险 |
| Chatbot 健康助手 | 可解释、自然语言 | 容易泛化、越界、缺少事实源 | 规则先决策，AI 只解释 |

## 2.2 模型与方案选型

### 任务要求

本项目对模型的要求不是“自由规划”，而是“在严格约束下解释已有决策”：

- 输入必须是脱敏聚合证据；
- 输出必须是结构化 JSON；
- 数字必须来自 allowedFacts；
- decision 必须与规则层一致；
- actionSuggestion 必须在 allowedActionSuggestions 白名单内；
- 不通过校验时 fallback。

### 推荐架构

```text
Fixed Workflow
+ Rule Decision Engine
+ Evidence Builder
+ Serverless LLM Explanation
+ Validator / Safety Gate
+ Cache / Fallback
+ Offline Eval
```

### 为什么选择这个方案

- 咖啡因残留和风险判断适合确定性公式与规则；
- 个性化解释适合 LLM 处理语言组织和证据选择；
- localStorage 足以支撑原型阶段的记录、反馈和记忆；
- Serverless API 可以保护 API Key，不把密钥放进浏览器；
- Hard Gate 可以把 AI 输出风险控制在用户可见前。

### 为什么暂不用更复杂方案

- 暂不用聊天入口：核心任务不是聊天，而是喝前决策；
- 暂不用复杂 Agent Team：链路固定，不需要多智能体规划；
- 暂不用向量数据库：第一版历史窗口是最近 14 天结构化证据；
- 暂不用真实 OCR：OCR 目前不是 AI 个性化解释的必要条件；
- 暂不用自动模型训练：没有真实用户样本，也不应自动调整健康建议。

## 2.3 主链路设计

### 主链路一句话

这条链路把“用户准备喝的这一杯 + 今日摄入状态 + 近期历史反馈”经过“规则判断”和“AI 个性化解释”，转成“现在这杯怎么喝”的可执行建议。

### L1-L8 链路地图

| 节点 | 输入 | 处理动作 | 输出 | 是否需要 AI | 状态沉淀 | 失败点 | Fallback | 用户确认边界 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| L1 触发喝前模拟 | 用户选择饮品、当前时间 | 进入模拟弹窗 | 模拟请求 | 否 | 无 | 饮品未选中 | 提示先选饮品 | 用户主动触发 |
| L2 读取规则材料 | 今日记录、用户设置、反馈、饮品 caffeineMg | 读取本次模拟所需数据 | records、settings、feedback、simulatedDrink | 否 | 无 | 设置缺失、饮品无 caffeineMg | 使用默认设置或要求补充 | 用户确认饮品 |
| L3 规则决策 | L2 材料 | 计算当前剩余、睡前残留、今日累计、风险等级 | RuleDecisionSnapshot | 否 | 可用于缓存 hash | 时间异常、数值异常 | 规则型保守建议 | 规则结论不需用户确认 |
| L4 证据压缩 | 最近 14 天记录、dailyStatusMemory、feedbackMemory、RuleDecisionSnapshot | 构建 CompactEvidenceContext、allowedFacts、evidenceIds、证据门槛 | 脱敏结构化证据 | 否 | dataVersionHash | 证据不足 | 不调用 LLM，展示规则解释 | 不发送个人自由文本 |
| L5 模型解释 | CompactEvidenceContext | 调用服务端 LLM，生成 JSON 解释 | explanation、historicalObservation、evidenceSummary、actionSuggestion | 是 | 可缓存模型输出 | 超时、格式错误、越界输出 | validation_failed / service_unavailable fallback | 模型不能改规则结论 |
| L6 校验与展示 | 模型输出、规则快照、allowedFacts | Schema、数字、证据、行动、安全校验；前端映射中文 | 可展示解释卡 | 否 | 记录 fallback 类型 | 内部枚举、非法数字、因果化 | 展示规则型解释 | 用户看到前必须校验 |
| L7 用户动作 | 记录这杯、半杯、低因、不喝、查看依据 | 用户选择下一步 | 行动结果或反馈 | 否 | 记录/反馈/Bad Case | 用户觉得不符合 | 保留规则建议，允许反馈 | 写入需用户确认 |
| L8 状态回流 | 新记录、反馈、解释反馈 | 更新日历、常喝、custom drink、Bad Case | 下次建议材料 | 否 | localStorage memory | 数据污染、模拟误入真实记录 | 数据隔离规则 | 不自动训练模型 |

## 2.4 功能模块与节点实现

### 2.4.1 饮品选择与记录

目标：降低“我不知道多少 mg”的记录门槛。

第一版能力：

- 饮品库搜索；
- 自定义饮品；
- 常喝饮品快捷复用；
- no_match 用户确认后保存；
- 记录确认弹窗只展示必要字段；
- 保存后刷新首页、曲线、状态日历和记忆。

关键规则：

- 搜索无结果不自动写入 no_match；
- 取消不进入长期记忆；
- pending no_match 不参与摄入计算；
- custom drink 真实记录后才进入常喝统计。

### 2.4.2 喝前模拟与规则决策

目标：回答“如果现在喝这杯，会怎么样？”

第一版输出：

- 这杯饮品咖啡因；
- 喝完后今日累计；
- 当前体内剩余；
- 睡前预计残留；
- 睡眠风险；
- action：整杯 / 半杯 / 低因 / 不喝。

规则层负责所有数值和结论。LLM 不能修改这些结果。

### 2.4.3 首页今日饮用建议

目标：用户一进来就知道今天怎么做。

首页默认展示：

- 今日饮用建议主卡；
- 豆豆状态；
- 今日已摄入；
- 睡前预计残留；
- 今日目标或还可摄入；
- 记录一杯；
- 喝前模拟；
- 最近记录。

首页不默认展示：

- 技术机制；
- 工具调用轨迹；
- 多个重复状态卡；
- 复杂趋势图；
- 聊天入口。

### 2.4.4 洞察页与状态日历

目标：帮助用户复盘近 7 天趋势和某一天发生了什么。

洞察页包含：

- 本周结论；
- 代谢曲线；
- 睡眠影响；
- 耐受 / 敏感；
- 状态日历入口。

状态日历不是普通历史列表，而是记忆系统的前台展示和反馈入口：

- 默认展示近 14 天简洁日历；
- 点击日期打开底部抽屉；
- 展示当天摄入、残留、风险、记录明细和反馈；
- 支持补录反馈。

### 2.4.5 反馈、记忆与个性化校准

反馈分为两类：

- 睡眠类反馈：默认归属到上一睡眠周期或前一摄入日；
- 即时不适反馈：心悸、焦虑、手抖、胃不舒服，归属反馈发生当天。

记忆类型：

| 记忆 | 作用 | 写入条件 |
| --- | --- | --- |
| dailyStatusMemory | 按天沉淀摄入、残留和风险 | 新增/删除记录、保存反馈、设置变化后重算 |
| feedbackMemory | 保存主观睡眠和不适反馈 | 用户主动填写 |
| FrequentDrinkMemory | 降低重复记录成本 | 真实记录达到阈值或用户手动固定 |
| NoMatchDrinkMemory | 记录用户确认过的未知饮品 | 用户选择仅保存名称 |
| custom drink | 用户确认咖啡因后的自定义饮品 | 用户补充 caffeineMg 后生成 |

### 2.4.6 AI 个性化喝前解释

目标：解释“为什么这次建议半杯 / 低因 / 不喝 / 可以喝”。

触发方式：

- 只在喝前模拟结果页；
- 用户主动展开“为什么这样建议？”；
- 第一次展开才请求；
- 同一 dataVersionHash 使用缓存；
- 修改模拟条件清除旧解释。

展示结构：

1. 规则说明；
2. 规则依据；
3. AI 个性化解释；
4. 近期记录参考；
5. 建议行动；
6. 合并底部说明。

不展示：

- full_cup / half_cup / low_caf / no_more_today 等内部枚举；
- 工具函数名；
- Prompt；
- 完整模型输出；
- API Key 或 header；
- 用户自由文本备注。

## 2.5 Prompt 管理

第一版只需要一类真实 LLM Prompt：

### P1：个性化喝前解释 Prompt

目标：根据规则层已经生成的结论和脱敏历史证据，生成短解释。

输入：

- RuleDecisionSnapshot；
- CompactEvidenceContext；
- allowedFacts；
- allowedActionSuggestions；
- evidenceIds；
- evidenceStrength；
- observationType；
- relationToDecision。

模型输出字段：

```json
{
  "decision": "full_cup | half_cup | low_caf | no_more_today",
  "explanation": "",
  "historicalObservation": null,
  "observationType": "low_risk_positive_pattern | evening_intake_pattern | high_residual_pattern | sleep_feedback_overlap | discomfort_overlap | insufficient_data | none",
  "evidenceIds": [],
  "evidenceSummary": "",
  "actionSuggestion": "full_cup | half_cup | low_caf | no_more_today",
  "confidence": "low | medium | high"
}
```

服务端固定注入：

- dataLimitation；
- safetyNote。

Prompt 必须强调：

1. 先说明当前建议来自规则结果；
2. 只使用与当前 decision 相关的历史证据；
3. 支持性证据说明支持关系；
4. 对照证据必须说明当前场景与历史场景差异；
5. 没有相关历史证据时 historicalObservation 返回 null；
6. 不得生成输入外数字；
7. 不得出现内部枚举；
8. 不得做医学、因果或保证性表达。

### 2.6.1 RuleDecisionSnapshot

规则层统一输出：

- bedTime；
- halfLifeHours；
- safeSleepResidualMg；

- ruleActionText；
- allowedActionSuggestions；
- afterTodayTotalMg；
- simulatedDrink；

### 2.6.2 CompactEvidenceContext

证据层统一输出：

- requestId；
- dataVersionHash；
- schemaVersion；
- ruleDecision；
- historicalEvidence；
- evidenceIds；
- allowedFacts；
- allowedActionSuggestions；
- constraints。

### 2.6.3 AI Response

最终 API Response 对前端返回：

- resultType：llm / fallback；
- fallbackType；
- validationFailureCode；
- validationFailureField；

- evidenceIds；

## 2.7 失败兜底与安全边界

### Fallback 类型

| 场景 | 处理 |
| --- | --- |
| 证据不足 | 不调用 LLM，展示规则解释和“近期重复证据不足”说明 |
| API 超时 / 网络错误 / 429 / 5xx | 最多技术性重试 1 次，仍失败则 fallback |
| Schema 错误 | 技术性格式问题可重试 1 次 |
| decision 不一致 | 不重试，直接 fallback |
| action 不在白名单 | 不重试，直接 fallback |
| 非法 evidenceId | 不重试，直接 fallback |
| 输入外数字 | 不重试，直接 fallback |
| 医学化 / 因果化 / 绝对化 | 不重试，直接 fallback |
| 内部枚举暴露 | 不重试，直接 fallback |

### Safety Gate

用户可见自然语言字段不得出现：

- 医学诊断；
- 治疗建议；
- 药物建议；
- “一定会失眠”等绝对承诺；
- “导致 / 造成 / 引发”等确定因果表达；
- full_cup、half_cup、low_caf、no_more_today 等内部枚举；
- allowedFacts 以外的数字。

### 数据隐私边界

外部模型只接收脱敏聚合字段。

不得发送：

- API Key；
- Authorization Header；
- LocalStorage key；
- 设备信息；
- 个人标识；
- 用户自由文本备注；
- 完整原始记录列表；
- 自定义饮品真实名称。

## 2.8 3-7 天最小证据闭环

第一版要证明产品不是空想，最少需要跑通以下证据：

1. 用户通过饮品库或常喝饮品完成记录；
2. 喝前模拟给出确定性规则结论；
3. 用户展开“为什么这样建议？”时，证据足够则请求 LLM；
4. LLM 结果通过校验后显示个性化解释；
5. 模型越界时 fallback 到规则解释；
6. 新增记录或反馈后，状态日历与后续证据变化；
7. Eval case 能复现正常、边界、fallback 和安全场景。

# 三、评测体系与迭代机制

这一部分回答怎么判断输出好坏，低分归因到哪一环，改完以后怎么复测。

## 3.1 评测定位与对象

评测目标不是证明模型“聪明”，而是证明 AI 解释在当前产品链路中：

- 不改写规则结论；
- 不编造数字；
- 不乱引用证据；
- 不制造医疗或因果风险；
- 能让用户更理解本次建议；
- 失败时能安全降级。

评测对象：

| 层级 | 评测对象 |
| --- | --- |
| L3 规则决策 | RuleDecisionSnapshot 数值一致性 |
| L4 证据压缩 | evidenceIds、allowedFacts、minimumEvidenceMet |
| L5a 模型与校验 | Schema、Validator、Fallback、Safety |
| L5b 前端展示 | 中文映射、缓存、折叠区、信息层级 |
| L7 用户反馈 | 有帮助 / 不符合 / 查看依据 |
| L8 Bad Case | 错误归因、复测和边界记录 |

## 3.2 评测集与 Eval case 字段

当前个性化解释评测集位于：

```text
evaluation/personalized_explanation/dataset.json
```

当前规模：

- 总 Case：38 条；
- decision 分布：full_cup 8、half_cup 12、low_caf 8、no_more_today 10；
- primary_case_type 分布：normal 5、confusing 2、boundary 18、frontend 5、fallback 5、preview 3。

Eval case 应包含：

- case_id；
- primary_case_type；
- risk_tags；
- decision；
- context；
- expected；
- required_text_signals；
- expected_validation_failure_code；
- expected_validation_failure_field；
- acceptable_sources；
- expected_user_final_safe。

## 3.3 Rubric 与 Hard Gate

### Hard Gate

以下问题必须硬拦截：

- Schema 不合法；
- decision 与规则层不一致；
- 输出包含输入外数字；
- evidenceId 非法；
- actionSuggestion 超出白名单；
- 医学化表达；
- 因果化表达；
- 绝对承诺；
- 内部枚举暴露；
- fallback 类型或说明错误。

Hard Gate 同时区分三件事：

1. 模型是否生成违规内容；
2. Validator 是否成功拦截；
3. 用户最终看到的结果是否安全。

### 质量评分

1-3 分只用于质量维度：

- 证据相关性；
- 解释清晰度；
- 简洁性；
- 行动性；
- 前端信息层级。

## 3.4 Bad Case、复测口径与表达边界

Bad Case 应记录：

- Case ID；
- 来源阶段；
- 用户输入或模拟场景；
- 前置数据；
- 期望结果；
- 实际结果；
- 错误模块；
- 错误类型；
- 严重度；
- 根因；
- 修复动作；
- 修改文件；
- 原用例复测结果；
- 全量回归结果；
- 当前状态。

当前项目已有 Stage 6 透明化复盘，记录了：

- Bad Case 总数：44；
- P0 / P1 / P2：7 / 36 / 1；
- 已修复：38；
- 未修复：6 个 P1；
- P0 Fix Rate：100%。

这些结果来自设计型离线评测集和本地/Preview 验证，不应包装成真实线上用户指标。

## 4.1 首页

首页回答：

> 今天还能不能喝？

模块结构：

1. Header：Caffeine Coach、你的咖啡因小教练、我的入口；

关键体验：

- 常喝饮品快捷复用；
- 饮品库优先；
- no_match 只在用户确认后进入记忆；
- 保存记录按钮固定在弹窗底部；

默认展示：

# 五、数据结构与状态管理

## 5.1 核心数据

| 数据 | 说明 | 事实来源 |
| --- | --- | --- |
| intake records | 用户真实摄入记录 | 用户保存记录 |
| drink library | 默认饮品库 | `src/data/drinks.json` |
| custom drinks | 用户自定义饮品 | 用户确认 caffeineMg |
| settings | 作息、敏感度、半衰期等 | 我的页设置 |

| no_match memory | 待补充饮品 | 用户确认保存名称 |

## 5.2 关键边界

- dailyStatusMemory 是派生快照，不是唯一事实来源；
- 修改记录、作息、敏感度、半衰期或反馈后，应基于原始数据重算；
- 敏感度校准不能自动修改用户设置；
- no_match 不能伪装成饮品库匹配；
- 睡眠反馈归属上一睡眠周期或前一摄入日；
- 即时不适反馈归属反馈当天。

## 5.3 数据刷新

统一事件：

- record_added；
- record_deleted；
- settings_updated；
- feedback_saved。

每次触发后刷新：

- 今日饮用建议；
- 当前残留；
- 睡前预计残留；
- 最近记录；
- 洞察统计；
- 状态日历。

# 六、权限与安全

## 6.1 Permission 分级

| 等级 | 操作 |
| --- | --- |
| allow | 读取记录、搜索饮品、计算残留、喝前模拟、生成建议、用户明确点击后的添加记录 |
| ask | 删除单条记录、更新影响建议的关键设置、fuzzy_match 饮品确认 |
| strong_confirm / deny | 删除全部记录、重置全部数据 |

## 6.2 删除确认

用户删除单条记录时必须二次确认：

- 标题：确认删除这条记录？
- 说明：删除后会影响今日摄入、残留计算和趋势统计；
- 按钮：取消 / 确认删除。

## 6.3 健康安全边界

产品只提供生活方式建议。

可以说：

- 可能影响睡眠；
- 风险偏高；
- 建议谨慎；
- 建议今晚先暂停摄入。

不可以说：

- 一定会失眠；
- 已经导致健康问题；
- 诊断疾病；
- 推荐药物或治疗方案；
- 给孕产、疾病、未成年人等特殊人群做具体医学建议。

# 九、作品集 / 面试表达建议

## 可以这样表达

我设计并实现了一个咖啡因饮用决策助手。它不是单纯记录咖啡因 mg，而是围绕“用户准备喝一杯时，应该喝整杯、半杯、低因还是今天不喝”这个任务，设计了从饮品选择、规则计算、喝前模拟、反馈记忆到 AI 个性化解释的完整闭环。

项目中，规则层负责所有数值和风险判断，LLM 只负责解释当前规则建议。为了避免 AI 越界，我设计了 allowedFacts、allowedActionSuggestions、evidenceIds、Schema 校验、数字忠实、证据引用、内部枚举拦截和安全表达校验。模型失败或证据不足时，产品会回退到规则型解释，确保用户最终看到的建议仍然安全可控。

## 谨慎表达

以下说法必须配合样本、版本和评测口径：

- 建议质量提升；
- AI 解释更准确；
- 模型表现提升；
- 用户更愿意采纳建议。

## 暂时不能写

除非后续有真实用户日志和线上实验，否则不能写：

- 帮助用户改善睡眠；
- 真实用户满意度提升；
- 留存率提升；
- 已服务 XX 用户；
- 达到专业健康顾问水平；
- 自动学习并持续优化模型。

# 十、PRD 自检

| 检查项 | 结论 |
| --- | --- |
| 是否围绕具体用户任务，而不是堆技术 | 是，主任务是“现在这杯怎么喝” |
| 是否区分规则 baseline 和 AI 增强层 | 是，规则做计算，AI 做解释 |
| 是否避免聊天化 | 是，不新增聊天入口 |
| 是否定义了 AI 不做什么 | 是，不重算、不改建议、不诊断 |
| 是否有状态回流 | 是，记录、反馈、日历、饮品记忆回流 |
| 是否有评测机制 | 是，Eval case、Hard Gate、Bad Case |
| 是否可做作品集表达 | 是，包含场景、链路、边界和验证方式 |
| 是否存在过度包装风险 | 已标注，不能宣称真实线上效果或健康改善 |

## 阶段结论

Caffeine Coach 当前已经从“咖啡因小工具”升级为一个有明确业务场景的 AI 决策解释原型。

它的核心竞争点不在于把所有功能 AI 化，而在于：

1. 用规则稳定完成高风险数值判断；
2. 用记忆系统降低记录成本并沉淀个人状态；
3. 用 LLM 在严格事实和安全边界内解释当前喝前建议；
4. 用评测体系持续发现和修复 Bad Case。

下一阶段如果继续推进，应优先围绕真实用户样本、浏览器验收、Demo 录屏、README 和作品集表达收口，而不是继续新增聊天、复杂 Agent 或更多页面。
