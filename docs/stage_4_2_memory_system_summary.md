# Stage 4.2 饮品记忆系统总结

## 1. 阶段名称

Stage 4.2：Drink Memory System 饮品记忆系统

子阶段：

- Stage 4.2.1：常喝饮品记忆
- Stage 4.2.2：no_match 记忆与自定义饮品沉淀

对应提交：

```text
feat: complete stage 4.2 drink memory system
```

## 2. 阶段目标

Stage 4.2 的目标是围绕“记录一杯”这条高频链路，降低用户重复搜索、重复输入和手动确认咖啡因含量的成本。

这一阶段没有改变咖啡因代谢计算、今日建议、状态日历和反馈归属逻辑，而是在饮品选择入口增加轻量记忆能力：

- 已经真实记录过的饮品，可以沉淀为常喝饮品。
- 饮品库找不到的输入，可以在用户确认后沉淀为待补充饮品或自定义饮品。
- 自定义饮品真实记录后，可以继续参与常喝饮品统计。

## 3. 为什么需要饮品记忆系统

Caffeine Coach 的核心链路是：

```text
选择/搜索饮品 → 确认咖啡因含量和时间 → 保存记录 → 首页更新今日状态 → 判断今天还能不能喝
```

在这个链路中，最大的记录阻力不是用户不知道“今天喝了什么”，而是：

- 每次都要重新搜索同一杯饮品；
- 饮品库覆盖不全时，用户不知道如何继续；
- 用户不一定知道咖啡因 mg，但愿意在确认后保存自己的常见饮品；
- 搜索失败不应被误当成真实饮用行为；
- 常喝推荐应来自真实记录，而不是搜索历史。

因此，Stage 4.2 的设计原则是：记忆只服务记录效率，不替代事实来源；只有用户明确确认的内容，才进入长期记忆。

## 4. 常喝饮品记忆设计

常喝饮品记忆的价值是降低用户重复记录成本。

### 数据结构

核心结构为 `FrequentDrinkMemory`，记录：

- 饮品名称
- 品牌 / 来源
- 规格
- 咖啡因含量
- 记录次数
- 最近记录时间
- 是否用户手动固定
- 来源：`auto` 或 `manual`

### 生成规则

自动常喝饮品基于真实摄入记录生成：

- 近 14 天同一饮品出现次数达到阈值；
- 或累计记录次数达到阈值；
- 优先基于 `drinkId` 判断同一饮品；
- 没有 `drinkId` 时，基于品牌、名称和咖啡因含量生成稳定 id；
- 同名但咖啡因含量不同的饮品不会被误合并。

### 用户控制

用户可以：

- 将当前选中饮品手动加入常喝；
- 从常喝饮品中移除；
- 点击常喝饮品一键再次记录。

手动加入的常喝饮品不依赖次数阈值；移除后会进入轻量排除列表，避免短时间内又自动出现。

## 5. no_match 记忆设计

no_match 记忆的价值不是记录所有搜索失败，而是在用户明确确认时，为饮品库覆盖不足提供兜底。

### 触发边界

当用户搜索饮品时：

- 命中默认饮品库：走原有选择流程。
- 命中自定义饮品：走自定义饮品记录流程。
- 无可信匹配：只展示轻量提示“是否创建为自定义饮品？”

搜索无结果本身不会写入记忆。

### 写入边界

只有用户明确选择以下操作时，才写入：

- 仅保存名称，稍后补充；
- 补充咖啡因并记录；
- 在喝前模拟中补充咖啡因并保存为自定义饮品。

用户点击取消时：

- 不写入 no_match；
- 不生成自定义饮品；
- 不生成 ignored 状态；
- 不污染记忆池。

### 状态流转

当前 no_match 只保留两个状态：

```text
pending → converted
```

- `pending`：用户保存了名称，但还未补充咖啡因。
- `converted`：用户补充咖啡因后，已转为自定义饮品。

Stage 4.2.2 已取消长期 `ignored` 状态。

## 6. 自定义饮品沉淀逻辑

当用户为 no_match 补充咖啡因含量后，系统会复用现有自定义饮品结构：

- `sourceType = user_custom`
- `confidence = user_confirmed`
- `isCustom = true`

不同入口的语义不同：

- 在“记录一杯”中，用户补充咖啡因后会保存为自定义饮品，并直接记录到今天。
- 在“喝前模拟”中，用户补充咖啡因后会保存为自定义饮品，并用于本次模拟，不新增真实摄入记录。

自定义饮品保存后会进入统一饮品库，后续可以被搜索、选择和记录。

## 7. 为什么取消 ignored 状态

早期方案曾考虑 `ignored` 状态，用于表示用户暂时忽略未知饮品。

最终取消的原因是：

1. 搜索失败不等于用户真实关心该饮品。
2. 取消是一次性交互，不应成为长期记忆。
3. ignored 会增加记忆池噪音，后续还需要恢复、清理、解释等额外设计。
4. 当前阶段目标是最小闭环，优先保证“用户确认后才写入记忆”。

因此 Stage 4.2.2 的最终规则是：

- 取消不写入记忆；
- 不生成 ignored 状态；
- 记忆池只保留 pending 和 converted。

## 8. 三类饮品记忆的关系

Stage 4.2 实际形成了三类饮品相关记忆：

### 8.1 常喝饮品记忆

来自真实饮用行为或用户手动固定。

它回答：

```text
用户经常喝什么？
```

### 8.2 no_match 记忆

来自用户确认后的匹配失败输入。

它回答：

```text
哪些饮品库没有覆盖，但用户愿意保存？
```

### 8.3 自定义饮品

来自用户补充咖啡因后的确认饮品。

它回答：

```text
用户已经确认过哪些饮品和咖啡因含量？
```

### 关系规则

- pending no_match 不进入常喝饮品。
- converted no_match 会转为 custom drink。
- custom drink 被真实记录后，才参与常喝饮品统计。
- custom drink 也可以被用户手动加入常喝。
- 常喝饮品应优先来自真实饮用行为，而不是搜索历史。

## 9. 记忆写入、调用、更新、删除规则

### 写入规则

- 新增真实记录后，常喝饮品统计可更新。
- 仅保存名称后，写入 pending no_match。
- 补充咖啡因后，写入 custom drink，并将 no_match 转为 converted。
- 取消搜索无结果提示，不写入任何记忆。

### 调用规则

- 记录一杯入口优先展示常喝饮品。
- 饮品搜索结果合并默认饮品和自定义饮品。
- 待补充饮品只展示 pending no_match。
- 喝前模拟可复用自定义饮品，但不会新增真实记录。

### 更新规则

- 重复保存相同 no_match 时，基于 `normalizedInput` 合并，更新 `count` 和 `lastSeenAt`。
- 常喝饮品基于真实 `drinks` 记录派生，新增或删除记录后会同步变化。
- 手动固定的常喝饮品不因记录减少而立即消失。

### 删除规则

- 删除真实记录后，常喝统计会重新派生。
- 从常喝移除会记录轻量排除标记，避免短时间重复出现。
- converted no_match 不再作为待补充饮品重点展示。

## 10. 产品业务价值

Stage 4.2 的价值不在于“记住更多东西”，而在于把用户记录饮品时的摩擦降下来：

- 对高频饮品：减少重复搜索，一键记录。
- 对未知饮品：不打断流程，给出可控兜底。
- 对自定义饮品：让用户确认过的咖啡因含量能被复用。
- 对数据质量：避免把搜索失败、取消操作误记为真实行为。
- 对后续推荐：常喝饮品来自真实记录，使状态判断和趋势统计更可信。

## 11. 当前边界

Stage 4.2 不做：

- OCR；
- 复杂语义匹配；
- 自动联网查询咖啡因；
- 后端数据库；
- 复杂饮品知识库；
- 推荐算法调整；
- 将搜索历史直接作为常喝依据；
- 长期 ignored 状态。

取消操作只是一种一次性交互，不进入长期记忆系统。

## 12. 待优化方向

后续可以继续优化：

- 自定义饮品管理页；
- 品牌 / 杯型 / 容量维度的自定义饮品编辑；
- 相似 no_match 的人工合并；
- OCR 识别失败后的 no_match 沉淀；
- 常喝饮品排序解释；
- 端到端自动化测试覆盖 no_match 交互；
- 饮品库覆盖率 Bad Case 看板。

## 13. 可用于简历的项目表达

### 精简版

- 为 Caffeine Coach 设计并实现轻量饮品记忆系统，基于真实摄入记录沉淀常喝饮品，并在饮品库无匹配时通过用户确认生成 no_match / 自定义饮品记忆，降低重复记录成本并保证数据来源可信。

- 设计“用户确认后才写入记忆”的 no_match 机制，避免将搜索失败或取消操作污染为长期记忆；自定义饮品只有在真实记录后才参与常喝统计，保证推荐入口来自真实饮用行为。

### 展开版

**背景：** Caffeine Coach 的核心链路是快速记录咖啡因摄入，并判断今天是否适合继续喝。用户在记录时经常遇到两个问题：常喝饮品需要反复搜索，饮品库没有覆盖的饮品又无法顺畅记录。

**问题：** 如果把所有搜索失败都记下来，会污染记忆池；如果把搜索历史直接作为常喝依据，又会让常喝推荐偏离真实饮用行为。产品需要一套轻量记忆机制，既降低记录成本，又保证数据可信。

**设计：** 我将饮品记忆拆成三类：常喝饮品、no_match 和自定义饮品。常喝饮品只基于真实记录或用户手动固定生成；no_match 只有在用户选择“仅保存名称”或“补充咖啡因并记录”时才写入；自定义饮品补充咖啡因后进入统一饮品库，并在真实记录后参与常喝统计。同时取消长期 ignored 状态，避免把一次性取消变成长期记忆。

**结果：** 用户可以在记录入口一键复用常喝饮品；饮品库找不到时，也能补充咖啡因并沉淀为自定义饮品；搜索无结果后取消不会污染记忆池。Stage 4.2 与状态日历、反馈记忆、首页建议和代谢曲线保持兼容。

**产品价值：** 这套机制把“搜索不到”和“经常喝”区分开，让记忆系统更贴近真实行为。它既降低了高频记录成本，也为后续饮品库补充、OCR 识别和个性化推荐提供了更可信的数据基础。

## 14. 阶段结论

Stage 4.2 完成了 Caffeine Coach 饮品记忆系统的最小闭环。

常喝饮品记忆解决了重复记录成本问题；no_match 记忆解决了饮品库覆盖不足时的兜底问题；自定义饮品沉淀让用户确认过的咖啡因含量能够被复用。

整个阶段保持了一个关键边界：记忆系统只记录用户明确确认或真实饮用后的信息，不把搜索失败和取消操作当作长期事实。Stage 4.2 已具备进入 Stage 4.3 的基础。
