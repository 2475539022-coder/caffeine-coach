# Caffeine Coach 阶段 4：记忆系统设计

## 0. 文档定位

本文件用于设计 Caffeine Coach 阶段 4 的轻量记忆系统。

这里的“记忆”不是聊天记忆，也不是让产品变成 AI 助手对话。它是围绕用户的咖啡因摄入、睡前残留、主观反馈和敏感度变化形成的行为记忆，用来让产品更稳定地回答：

- 今天还能不能喝？
- 为什么今天建议更保守？
- 最近是不是经常睡前残留偏高？
- 我的咖啡因敏感度有没有变化？
- 哪些饮品是我常喝、常查、常需要手动确认的？

记忆系统必须服务当前产品核心链路：

选择 / 搜索饮品 → 确认咖啡因含量和时间 → 保存记录 → 首页更新今日状态 → 判断今天还能不能喝

## 1. 记忆系统目标

记忆系统的目标不是生成更多内容，而是减少用户重复输入、提高建议稳定性，并让趋势复盘更有上下文。

阶段 4 的目标包括：

1. 记住用户常喝饮品，降低记录门槛。
2. 记住 no_match 饮品，帮助后续补库和减少错误匹配。
3. 记住每日摄入、睡前残留、风险等级和反馈，形成按天沉淀的状态。
4. 记住近期反馈摘要，让推荐量和敏感度校准更连续。
5. 记住用户对解释深度的偏好，避免每次都展示过多说明。
6. 让“状态日历”成为记忆系统的前台展示和反馈入口。

## 2. 需要记住哪些内容

阶段 4 只记住与主链路有关的数据，不做泛化个人画像。

### 2.1 常喝饮品记忆

记录用户经常选择、搜索、确认或手动添加的饮品。

内容包括：

- 饮品名称
- 品牌
- 杯型 / 容量
- 用户确认后的咖啡因含量
- 最近记录时间
- 使用次数
- 是否用户自定义

用途：

- 记录一杯时优先展示更相关候选。
- 喝前模拟时减少搜索成本。
- no_match 后再次输入相似饮品时提供更好的候选。

### 2.2 no_match 记忆

记录饮品库没有可信匹配的输入，不编造咖啡因含量。

内容包括：

- 原始输入文本
- 用户补充的品牌 / 杯型 / 容量
- 用户最终手动输入的咖啡因含量
- 是否后来被保存为常喝饮品
- 最近出现时间
- 出现次数

用途：

- 帮助后续补库。
- 再次搜索相同饮品时提示“上次你手动记录为 xxmg”。
- 评估饮品库覆盖不足的 Bad Case。

### 2.3 每日状态记忆

按天沉淀用户当天的摄入和风险状态。

内容包括：

- 日期
- 当日总摄入量
- 当日记录次数
- 最晚摄入时间
- 睡前预计残留
- 睡眠风险等级
- 是否超过今日推荐量
- 是否出现晚间摄入
- 当天豆豆状态
- 当天核心建议摘要

用途：

- 状态日历展示每天的状态。
- 洞察页计算 7 日趋势。
- 首页在缺少今日反馈时参考近期风险模式。

### 2.4 主观反馈记忆

记录用户每天主动反馈的感受。

内容包括：

- 提神效果评分
- 是否心慌 / 焦虑 / 胃不舒服 / 手抖
- 昨晚睡眠质量
- 入睡速度
- 是否觉得越喝越没用
- 反馈时间

用途：

- 调整敏感度判断。
- 判断耐受 / 敏感趋势。
- 让状态日历可以补录或回看当天感受。
- 给用户解释“为什么今天建议更保守”。

### 2.5 敏感度校准记忆

记录系统如何根据问卷、设置和反馈逐步校准用户对咖啡因的反应。

内容包括：

- 用户当前敏感程度
- 当前推荐策略：宽松 / 平衡 / 严格
- 近期不适反馈次数
- 近期睡眠差次数
- 当前睡前更安心目标
- 最近一次校准原因

用途：

- 首页今日饮用建议。
- 喝前模拟风险判断。
- 洞察页“耐受 / 敏感”解释。

### 2.6 解释偏好记忆

记录用户是否经常展开“为什么这样建议”。

内容包括：

- 是否偏好简短建议
- 是否经常查看原因和依据
- 是否经常查看科普折叠卡

用途：

- 首页默认保持简短。
- 用户经常查看解释时，可以在展开区给更完整的依据。
- 不在主界面默认堆叠复杂说明。

## 3. 每类记忆的数据来源

| 记忆类型 | 数据来源 | 是否需要用户确认 |
| --- | --- | --- |
| 常喝饮品记忆 | 记录一杯、喝前模拟、我的常喝饮品 | 保存记录后自动更新使用次数不需要确认；创建自定义饮品、沿用 no_match 手动值、保存为常喝饮品时需要确认 |
| no_match 记忆 | 饮品搜索、OCR 匹配失败、手动输入 | 保存手动结果需要确认 |
| 每日状态记忆 | 摄入记录、设置、半衰期计算、睡前残留计算 | 自动生成，可被反馈补充 |
| 主观反馈记忆 | 我的页反馈、状态日历补录 | 用户主动填写 |
| 敏感度校准记忆 | 设置问卷、反馈、摄入与睡眠残留 | 自动更新，但应可解释 |
| 解释偏好记忆 | 展开解释、查看科普、关闭说明 | 自动记录，低优先级 |

## 4. 每类记忆如何更新

### 4.1 新增记录后

触发：

- 更新今日摄入量。
- 更新今日睡前预计残留。
- 更新每日状态记忆。
- 更新常喝饮品使用次数。
- 更新状态日历当天状态。
- 刷新首页今日饮用建议。
- 刷新洞察页 7 日趋势。

### 4.2 删除记录后

触发：

- 重算今日摄入量。
- 重算睡前预计残留。
- 重算每日状态记忆。
- 更新状态日历当天状态。
- 刷新首页和洞察页。

删除必须保留二次确认，因为它会改变历史趋势和今日建议。

### 4.3 保存反馈后

触发：

- 更新当天主观反馈记忆。
- 更新敏感度校准记忆。
- 更新每日状态记忆里的反馈状态。
- 更新状态日历当天标记。
- 刷新首页今日饮用建议。
- 刷新洞察页耐受 / 敏感趋势。

### 4.4 no_match 后手动确认

触发：

- 保存 no_match 原始输入。
- 保存用户确认的咖啡因含量。
- 可提示“是否保存为常喝饮品”。
- 后续相同搜索优先提醒用户上次确认值。

### 4.5 修改设置后

触发：

- 更新敏感度校准记忆。
- 重算今日推荐量。
- 重算睡前残留目标相关判断。
- 更新今日状态。
- 必要时提示用户“后续建议会按新设置计算”。

## 5. 每类记忆用在哪里

### 5.1 首页

首页使用：
首页今日建议优先基于今日记录和用户设置实时计算；dailyStatusMemory 只作为近期模式参考，不作为首页今日建议的唯一数据源。

- 今日状态记忆
- 最近反馈记忆
- 敏感度校准记忆
- 今日记录

首页不展示完整历史，只使用记忆帮助生成更稳定的今日建议。

### 5.2 记录页

记录页使用：

- 常喝饮品记忆
- no_match 记忆
- 最近记录

目标是减少搜索和手动输入成本。

### 5.3 喝前模拟

喝前模拟使用：

- 常喝饮品记忆
- 今日状态记忆
- 敏感度校准记忆
- 当前睡前残留

目标是判断“如果现在喝这一杯，会不会影响睡眠”。

### 5.4 洞察页

洞察页使用：

- 每日状态记忆
- 主观反馈记忆
- 敏感度校准记忆

目标是做近 7 天趋势复盘，而不是复杂健康报告。

### 5.5 我的页

我的页使用：

- 咖啡因习惯设置
- 主观反馈
- 常喝饮品
- 数据管理入口

它是记忆的管理和校准入口，但不应变成参数配置页。

## 6. 状态日历在记忆系统中的作用

状态日历不是普通历史记录页。

它是记忆系统的前台展示和反馈入口，用于按天沉淀用户的摄入、风险和反馈数据。

状态日历首版作为洞察页中的二级入口，不默认占据洞察页大面积空间，不与趋势图同时堆叠展示。

状态日历的核心价值：

1. 把“每天发生了什么”可视化。
2. 让用户快速看到哪几天摄入偏高、睡前残留偏高或反馈不佳。
3. 允许用户补录某一天的主观反馈。
4. 让洞察页的 7 日趋势有可回溯的来源。
5. 让首页建议能引用近期模式，而不是只看当天记录。

状态日历按天展示的内容：

- 日期
- 当日总摄入
- 睡前预计残留
- 睡眠风险等级
- 豆豆状态
- 是否晚间摄入
- 是否超过推荐量
- 是否填写反馈

状态日历支持的轻操作：

- 查看某一天摘要
- 补充当天反馈
- 查看当天记录
- 跳转到记录页查看详情

状态日历不做：

- 不展示复杂报告
- 不做医疗化判断
- 不替代记录页
- 不提供聊天入口
- 不展示底层工具调用轨迹

## 7. 状态日历与洞察页、首页、记录页的关系

### 7.1 与首页的关系

首页回答“今天还能不能喝”。

状态日历回答“最近每天的状态是什么样”。

首页可以使用状态日历沉淀的数据，但不应把日历展开到首页。首页最多保留轻入口，例如“查看最近状态”。

### 7.2 与洞察页的关系

洞察页做趋势复盘。

状态日历是趋势背后的日粒度数据入口。

建议关系：

- 洞察页展示 7 日摄入趋势、睡前残留趋势、耐受 / 敏感趋势。
- 状态日历展示每一天的状态。
- 用户在洞察页看到异常天数后，可以进入状态日历查看具体是哪几天。

### 7.3 与记录页的关系

记录页是饮品记录明细。

状态日历是按天聚合后的状态视图。

用户如果想知道某天喝了哪些饮品，应从状态日历跳转到记录页的当天分组。

### 7.4 与我的页的关系

我的页是设置和反馈入口。

状态日历可以承接“补录某天反馈”，但长期设置仍属于我的页。

## 8. MVP 实现范围

阶段 4 MVP 建议只做以下范围：

1. 新增每日状态记忆结构，仍保存在 localStorage。
2. 新增状态日历入口，建议放在洞察页内，而不是首页第一屏。
3. 状态日历展示近 14 天或本月简化视图。
4. 每一天显示风险颜色和是否有反馈。
5. 点击某一天打开轻量详情：
   - 总摄入
   - 睡前预计残留
   - 睡眠风险
   - 当天记录数量
   - 反馈摘要
6. 支持从某一天详情补充主观反馈。
7. no_match 记忆、常喝饮品排序、补库入口不进入阶段 4.1，放到阶段 4.2 实现。

MVP 不需要新后端，不需要登录，不需要真实 LLM，不需要跨设备同步。

## 9. 暂不做的内容

阶段 4 暂不做：

1. 聊天记忆。
2. 跨设备同步。
3. 云端数据库。
4. 医疗诊断或健康评分。
5. 自动推断疾病、焦虑障碍或睡眠障碍。
6. 长期复杂报告。
7. 自动删除或自动修改用户历史记录。
8. 未经确认自动写入饮品记录。
9. 大模型自由改写风险判断规则。
10. 将 Agent Loop、Tool Use、Hooks 等技术词展示给普通用户。

## 10. 后续实现顺序

### 第一步：整理记忆数据结构

定义本地记忆结构：

- `dailyStatusMemory`
- `noMatchMemory`
- `drinkUsageMemory`
- `feedbackMemory`
- `preferenceMemory`

先只落 localStorage，不引入后端。

### 第二步：统一写入时机

在现有事件后更新记忆：

- `record_added`
- `record_deleted`
- `feedback_saved`
- `settings_updated`
- OCR 确认记录

沿用当前 Hooks 思路，只做刷新和记忆沉淀，不做复杂插件系统。

### 第三步：新增状态日历 MVP

优先放在洞察页，作为“趋势复盘”的下钻入口。

首版只展示近 14 天或本月状态，不进入首页第一屏。

### 第四步：补录反馈闭环

用户点击某一天后，可以补充当天反馈。

补录后更新：

- 当天反馈记忆
- 敏感度校准
- 洞察页趋势
- 首页后续建议

### 第五步：常喝饮品排序

根据使用次数、最近使用时间、用户自定义状态，优化饮品选择排序。

此步骤只影响候选排序，不改变饮品匹配的可信规则。

### 第六步：no_match 补库入口

将最近 no_match 饮品放在“我的”或“记录”页的轻量管理区。

用户可以把它保存为常喝饮品，或补充品牌、杯型、咖啡因含量。

### 第七步：偏好化解释深度

如果用户经常展开解释，可以在“为什么这样建议？”里展示更完整依据。

如果用户很少展开解释，首页继续保持一句结论 + 一个行动建议。

## 11. 验收标准

阶段 4 记忆系统完成后，应满足：

1. 新增 / 删除记录后，每日状态记忆会更新。
2. 保存反馈后，状态日历当天会显示已反馈。
3. 洞察页可以基于每日状态记忆展示 7 日趋势。
4. 状态日历能展示每天的摄入、残留和风险状态。
5. no_match 不会编造饮品数据，只沉淀用户确认后的信息。
6. 常喝饮品可以根据使用记忆减少记录成本。
7. 首页仍然简洁，不变成历史报告页。
8. 用户主界面不出现 Agent Loop、Tool Use、Hooks 等技术词。
9. 所有写入类动作仍需要用户明确确认。
10. 所有健康表达保持克制，不做医疗诊断。

## 12. 关键边界与补充规则

### 12.1 `dailyStatusMemory` 是派生快照，不是唯一事实来源

`dailyStatusMemory` 只用于日历和洞察页展示，不替代原始摄入记录、用户设置和反馈记录。

当用户修改记录、作息时间、敏感度、半衰期或反馈后，应基于原始数据重新计算每日状态。

### 12.2 敏感度校准不能自动修改用户设置

系统可以生成 `suggestedSensitivityLevel` 和 `calibrationReason`，用于解释为什么建议更保守。

但不能未经用户确认自动修改 `userSensitivitySetting`。

如需修改敏感度设置，必须通过用户确认。

### 12.3 `no_match` 记忆不能伪装成饮品库匹配

`no_match` 历史只能作为 `user_history` 或 `user_confirmed` 来源展示。

再次出现相同输入时，应提示“上次你手动记录为 xxmg，是否沿用”，不能作为 `exact_match` 或 `fuzzy_match` 返回。

### 12.4 睡眠反馈归属规则

睡眠类反馈默认归属到上一睡眠周期或前一摄入日。

即时不适反馈，如心悸、焦虑、手抖，归属到反馈发生当天。

### 12.5 页面边界规则

首页回答今天怎么做。

洞察页回答最近趋势如何。

状态日历回答某一天发生了什么。

记录页回答喝了什么。

我的页负责设置和数据管理。

状态日历不重复展示洞察页完整趋势图，洞察页不展开日历每日明细。

### 12.6 MVP 拆分

阶段 4.1 优先实现：

- `dailyStatusMemory`
- `feedbackMemory`
- 状态日历
- 反馈补录

阶段 4.2 再实现：

- `drinkUsageMemory`
- `noMatchMemory`
- 常喝饮品排序
- 补库入口
