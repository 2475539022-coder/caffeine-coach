# Caffeine Coach 产品范围护栏

## 1. 产品定位

Caffeine Coach 是一个轻量咖啡因记录与饮用建议工具。

它帮助用户快速记录咖啡因摄入，并判断今天是否适合继续摄入，重点关注睡眠影响。

产品不以聊天、复杂健康分析或专业医疗判断为核心。前台体验应始终围绕“今天还能不能喝、怎么快速记录、为什么这样建议”展开。

## 2. 核心链路

所有用户主界面改动都应服务以下链路：

选择 / 搜索饮品 → 确认咖啡因含量和时间 → 保存记录 → 首页更新今日状态 → 判断今天还能不能喝

这条链路优先级高于展示机制、技术说明和复杂解释。

## 3. 必须保留的核心功能

1. 记录一杯
2. 喝前模拟
3. 今日摄入量
4. 睡前预计残留
5. 睡眠风险
6. 今日饮用建议
7. 最近记录
8. 近 7 天摄入趋势与睡前残留趋势

这些功能是 Demo 的基本产品骨架。后续前台优化不得弱化、隐藏或移除它们。

## 4. 明确不做

1. 不做聊天机器人
2. 不做复杂健康报告
3. 不做医疗诊断
4. 不做无数据支撑的提神变化预测
5. 不在用户主界面展示 Agent Loop、Tool Use、Hooks 等技术词

底层可以保留 Agent Loop、Tool Use、Permission、Hooks 等机制，但它们应作为产品能力支撑，不应成为普通用户看到的主卖点。

## 5. 功能准入标准

任何新增或修改的模块，必须说明它服务核心链路中的哪一步：

- 记录
- 判断
- 建议
- 反馈
- 趋势复盘

如果一个模块不能服务上述任一环节，不应放进用户主界面。

## 6. 前台表达原则

1. 首页优先回答“今天还能不能喝”。
2. 高频入口优先，包括“记录一杯”和“喝前模拟”。
3. 数据解释默认收起，用户主动查看时再展开。
4. 技术机制只放在开发调试或文档中，不进入用户默认视图。
5. 文案应生活化、克制、可执行，不制造健康焦虑。
6. 健康相关表达只做生活方式建议，不做诊断或确定性判断。

## 7. 页面职责边界

### 今天

服务“判断”和“建议”。

必须优先展示今日饮用建议、关键摄入数据、睡眠风险、记录入口和喝前模拟入口。

### 记录

服务“记录”。

必须优先支持选择 / 搜索饮品、确认咖啡因含量和时间、保存记录、查看最近和历史记录。

### 洞察

服务“趋势复盘”。

展示近 7 天摄入趋势、睡前残留趋势、代谢曲线和轻量科普。不要变成复杂健康报告。

### 我的

服务“反馈”和个性化基础设置。

承载作息、管理目标、咖啡因习惯、今日感受、我的常喝饮品和数据说明。

## 8. 变更检查清单

每次前台改动前，先回答：

1. 这个改动服务核心链路中的哪一步？
2. 它是否让“记录一杯”更快或更清楚？
3. 它是否让“今天还能不能喝”更容易判断？
4. 它是否减少了重复信息，而不是增加新的解释负担？
5. 它是否避免在用户主界面暴露技术词？

如果以上问题无法回答清楚，应暂缓进入主界面。

## 9. 记录一杯交互规则

记录模块只负责一件事：快速找到饮品，确认信息，保存记录。

标准记录流程：

打开记录一杯 → 搜索 / 拍照 / 手动输入 → 系统返回候选 → 用户选择饮品 → 确认咖啡因含量和摄入时间 → 保存记录 → 首页状态更新

记录一杯支持三种入口：

- 搜索输入：输入品牌、饮品名、别名或关键词。
- 拍照识别：识别饮品包装、菜单、小票或截图。
- 手动输入：饮品库查不到时，用户手动输入名称和咖啡因含量。

记录页不应默认铺开完整饮品库列表。系统应根据用户输入推荐候选，而不是一开始展示所有数据。

候选卡片展示：

- 饮品名称
- 品牌
- 咖啡因含量
- 来源：品牌库 / 食品库 / 估算 / 用户自定义
- 置信度：高 / 中 / 低
- 杯型或容量信息

确认记录弹窗默认只展示：

- 饮品名称
- 咖啡因含量
- 摄入时间
- 保存记录按钮

以下内容放入“高级调整”，默认收起：

- 手动修改咖啡因含量
- 备注
- 置信度详细说明
- 饮品类型
- 来源说明
- 杯型补充

保存按钮必须固定在底部，用户不应滑到最底部才能保存。

## 10. 饮品匹配规则

饮品搜索返回三种状态：

- `exact_match`：明确匹配，可直接进入确认记录。
- `fuzzy_match`：模糊匹配，需要用户确认候选。
- `no_match`：无可信匹配，不允许编造，要求手动输入或补充信息。

当饮品无可信匹配时：

1. 不返回伪候选。
2. 不编造咖啡因含量。
3. 不继续计算风险。
4. 提示用户补充品牌、杯型或手动输入咖啡因 mg。
5. 可记录到 no_match 历史中，后续用于补库。

置信度只作为辅助信息，不应抢占主流程：

- 高置信：轻量标签。
- 中置信：轻量标签，可提示确认。
- 低置信：必须允许手动修改咖啡因含量。

## 11. 今日推荐量和睡眠风险表达

今日推荐摄入量不是固定值，应由基础建议量与个体校准共同决定。

用户不需要看到复杂公式。前台解释应表达为：

“今日推荐量根据你的睡眠时间、敏感度和近期反馈估算。”

可解释依据包括：

- 用户设置
- 睡眠时间
- 咖啡因敏感度
- 最近摄入记录
- 最近反馈
- 睡前残留风险

睡眠风险分为：

- 低风险：睡前残留较低，可正常摄入或少量摄入。
- 中风险：睡前残留可能偏高，建议控制分量。
- 高风险：睡前残留明显偏高，建议停止摄入。

健康表达必须克制：

可以说：

- 可能影响睡眠
- 风险偏高
- 建议谨慎
- 建议今晚先暂停摄入

不可以说：

- 一定会失眠
- 你已经咖啡因中毒
- 你有健康问题
- 你必须就医

如用户反馈严重不适，例如胸痛、严重心悸、呼吸困难，应建议停止继续摄入并咨询专业人士。

## 12. 咖啡豆状态规则

咖啡豆状态是前台拟人化表达层，用于降低产品冰冷感。它不直接做健康判断。

当前版本建议状态语义收敛为：

- 清醒豆：当前状态稳定，可以正常或少量摄入。
- 纠结豆：还能喝，但建议控制分量。
- 焦虑豆：负荷偏高，建议停止摄入。
- 恢复豆：已进入恢复期，建议不再摄入。
- 低因豆：可选择低因、半杯、小杯方案。

状态优先级：

焦虑豆 > 恢复豆 > 纠结豆 > 低因豆 > 清醒豆

AI 可以参与状态文案润色和解释组织，但不应自由决定状态类型、编造风险原因或输出医疗诊断。

## 13. 洞察页规则

洞察页回答：

“我最近的咖啡因摄入和睡眠风险有什么规律？”

洞察页不回答：

- 我是不是身体有问题？
- 我是不是咖啡因成瘾？
- 我未来一定会怎样？

洞察页核心模块：

- 近 7 天摄入趋势：每日摄入量、超过推荐量天数、晚间摄入次数。
- 睡前残留趋势：每日睡前预计残留、高风险天数、与睡眠时间的关系。
- 耐受 / 敏感度趋势：当前敏感度等级、当前推荐量为何调整、最近反馈如何影响推荐量。
- 反馈关联：睡眠差次数、心悸次数、焦虑次数、与晚间摄入的简单关联。

洞察页不应默认展示：

- 无反馈数据支撑的提神变化曲线
- 医疗化分析
- 复杂诊断结论
- 和咖啡因摄入无关的健康状态判断

## 14. Permission 与 Hooks 边界

Permission 用于控制用户数据是否会被误删、误改、误写入。

权限分级：

- `allow`：读取记录、搜索饮品、计算残留、喝前模拟、生成建议、用户明确点击保存记录。
- `ask`：删除单条记录、修改关键设置、fuzzy_match 饮品确认。
- `strong_confirm`：清空全部记录、重置全部数据。
- `deny`：未经用户确认的自动删除 / 自动重置。

Hooks 当前只做状态刷新，不做复杂插件系统。

支持事件：

- `record_added`：保存记录后，刷新首页建议、今日摄入、最近记录、洞察。
- `record_deleted`：删除记录后，刷新首页建议、今日摄入、最近记录、洞察。
- `settings_updated`：修改作息、敏感度、目标后，刷新首页建议、睡眠风险、洞察。
- `feedback_saved`：保存今日反馈后，刷新首页建议、洞察、敏感度趋势。

当前阶段暂不实现复杂插件系统、自动 bad case 记录或真实 LLM tool calling。

## 15. Bad Case 记录模板

以下情况需要记录 Bad Case：

- 未知饮品被错误匹配
- 系统给出不存在的咖啡因 mg
- 睡前残留高但仍建议继续喝
- 输出医疗化结论
- 同一指标重复出现多次
- 保存后首页没有刷新
- 删除后数据仍存在或建议未刷新

模板：

```md
### Bad Case 标题

- 日期：
- 页面 / 场景：
- 用户输入 / 操作：
- 期望结果：
- 实际结果：
- 调用工具：
- 是否涉及 no_match：
- 是否出现编造数据：
- 是否出现过度健康判断：
- 严重程度：P0 / P1 / P2 / P3
- 初步原因：
- 修复建议：
- 复测结果：
```

## 16. 当前最高优先级

当前最优先的工作不是继续加新功能，而是：

1. 收敛首页重复信息。
2. 重构记录一杯为搜索 / 拍照 / 手动输入三入口。
3. 恢复并明确耐受 / 敏感度趋势。
4. 明确咖啡豆状态规则。
5. 补充删除记录二次确认。
6. 用轻量 Hooks 保证记录、删除、反馈后状态刷新。
