# Stage 4.2.2：no_match 记忆与自定义饮品沉淀

## 阶段目标

本阶段让 Caffeine Coach 在饮品库没有匹配结果时，不再只是提示失败，而是先询问用户是否创建为自定义饮品。只有用户主动选择保存名称或补充咖啡因时，才会把该输入沉淀为「待补充饮品」或自定义饮品，后续继续用于记录、搜索和常喝饮品候选。

这一阶段仍然服务核心链路：选择/搜索饮品 → 确认咖啡因含量和时间 → 保存记录 → 首页更新今日状态。

## 新增能力

- 搜索不到饮品时，只展示创建自定义饮品的轻量提示，不自动写入记忆。
- 用户主动保存未知饮品后，生成 no_match 记忆。
- 用户重复保存同一未知饮品时，合并到同一条记忆并增加出现次数。
- 支持仅保存名称，稍后补充。
- 支持补充咖啡因含量并转为自定义饮品。
- 支持取消本次提示，取消不会写入 no_match，也不会生成 ignored 状态。
- 转为自定义饮品后，可被后续搜索和记录复用。
- 自定义饮品真实记录次数足够后，可进入常喝饮品候选。

## 数据结构说明

新增本地记忆结构 `NoMatchDrinkMemory`，存储在 localStorage 中。

```ts
type NoMatchDrinkMemory = {
  id: string;
  rawInput: string;
  normalizedInput: string;
  firstSeenAt: string;
  lastSeenAt: string;
  count: number;
  status: "pending" | "converted";
  caffeineMg?: number;
  convertedDrinkId?: string;
  note?: string;
};
```

`NoMatchDrinkMemory` 只表示「饮品库未命中的输入」，不等同于常喝饮品，也不替代自定义饮品库。

## no_match 触发规则

当用户在饮品选择入口搜索饮品时：

- 命中默认饮品库：继续原有选择流程。
- 命中自定义饮品：继续原有自定义饮品记录流程。
- 没有可信候选：展示轻量提示「没有找到该饮品，是否创建为自定义饮品？」。
- 用户选择「补充并保存」或「仅保存名称」时，才写入 no_match 或自定义饮品。
- 用户选择「取消」时，不写入 no_match，不生成 ignored 状态，不污染记忆池。

系统不会为未知饮品编造咖啡因含量。

## 标准化与合并规则

本阶段只做基础标准化：

- 去除首尾空格；
- 合并多个空格；
- 英文大小写归一；
- 去除常见标点；
- 中文不做复杂分词。

如果 `normalizedInput` 相同，则合并为同一条 no_match 记忆，更新 `count` 和 `lastSeenAt`，保留 `firstSeenAt`。

本阶段不做语义合并，例如「楼下拿铁」和「公司楼下拿铁」仍视为不同输入。

## 状态流转规则

```text
pending → converted
```

### pending

表示该饮品暂未匹配，等待用户补充或处理。

### converted

表示用户已经补充咖啡因含量，并将其转为自定义饮品。

转为 converted 后：

- 生成自定义饮品；
- 保存 caffeineMg；
- 记录 convertedDrinkId；
- 从待补充饮品主列表隐藏；
- 后续搜索该名称可以命中自定义饮品。

## 自定义饮品沉淀规则

用户为 no_match 补充咖啡因含量后，系统复用当前已有自定义饮品结构保存：

- `sourceType = user_custom`
- `confidence = user_confirmed`
- `isCustom = true`

自定义饮品保存后会进入统一饮品库来源，后续可被搜索、选择和记录。真实记录次数足够后，也会被 Stage 4.2.1 常喝饮品记忆识别为候选。

在「记录一杯」入口中，用户选择补充咖啡因后会直接保存为自定义饮品并记录到今天；在「喝前模拟」入口中，则保存为自定义饮品并用于继续模拟。

## 与常喝饮品记忆的关系

- no_match 只是匹配失败记忆，不直接成为常喝饮品。
- converted 后的自定义饮品需要被真实记录，才可能进入常喝饮品候选。
- 取消操作不进入记忆系统，也不进入常喝饮品候选。
- converted 的 no_match 不再出现在待补充饮品主列表，避免和自定义饮品重复展示。

## 验收通过项

- 输入饮品库不存在的名称时，会出现创建自定义饮品提示，但不会自动写入 no_match。
- 可仅保存名称，生成 pending 记忆。
- 重复保存同一未知饮品会合并次数，不新增重复项。
- 可补充咖啡因含量并转为自定义饮品。
- converted 后可在饮品搜索中复用。
- 取消操作不会产生 no_match 或 ignored 记录。
- 不影响原有饮品库命中流程。

## Stage 4.1 / 4.2.1 回归结果

- 新增记录仍走原有保存流程，并触发状态日历重算。
- 删除记录仍保留二次确认与状态重算。
- 点击某一天仍可查看当天记录明细。
- 睡眠反馈与即时不适反馈的归属规则未改动。
- 常喝饮品生成、固定、移除和一键记录逻辑未重构。
- 自定义饮品真实记录后仍可进入常喝饮品候选。

### 当前边界

- 本阶段不做复杂语义匹配。
- 本阶段不做联网查询咖啡因含量。
- 本阶段不做 OCR。
- 本阶段不做后端数据库持久化。
- 本阶段不做长期 ignored 状态。
- 取消操作仅作为一次性交互，不进入记忆系统。

### 待优化项

- 后续可支持更智能的相似饮品合并。
- 后续可支持用户管理自定义饮品。
- 后续可支持品牌/规格维度的自定义饮品管理。
- 后续可支持从 OCR 或图片识别中补充饮品信息。
- 后续可补充端到端自动化脚本，覆盖 no_match 的浏览器交互路径。

### 阶段结论

Stage 4.2.2 通过本轮验收。新的 no_match 逻辑符合“用户确认后才写入记忆”的产品边界：搜索无结果只提示，不自动沉淀；取消不会污染记忆池。记录入口和喝前模拟入口的语义已经区分，前者可补充并记录，后者只用于模拟。Stage 4.1 状态日历与 Stage 4.2.1 常喝饮品能力保持兼容，可以进入下一阶段。

## 当前边界

本阶段不做：

- OCR；
- 复杂语义匹配；
- 自动联网查询咖啡因；
- 大规模饮品库扩充；
- 复杂知识库；
- 后端数据库；
- 大范围 UI 重构；
- 推荐算法调整。

Stage 4.2.2 完成了匹配失败饮品的最小记忆闭环：未知饮品能被记住、合并、补充咖啡因，并沉淀为可复用的自定义饮品，同时不破坏 Stage 4.1 状态日历与 Stage 4.2.1 常喝饮品记忆。
