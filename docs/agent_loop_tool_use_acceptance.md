# Caffeine Coach Agent Loop + Tool Use 产品化接入验收文档

## 0. 验收结论

当前版本已从“设计说明 + 验收标准”进入实际验收阶段 v1.1。Agent Loop 和 Tool Use 已经不是孤立技术模块，而是作为底层建议机制接入到首页决策、喝前模拟和推荐解释中。

| 验收项 | 结论 | 说明 |
| --- | --- | --- |
| 首页今日饮用建议 | 通过 | 默认展示结论、关键指标和建议行动，不把页面做成聊天页。 |
| 为什么这样建议 | 通过 | 点击后展开原因和依据数据，支持用户理解推荐逻辑。 |
| 喝前模拟 | 通过 | 可结合饮品信息、今日摄入和作息估算风险，并展示行动建议。 |
| no_match 兜底 | 通过 | 未知饮品不返回伪候选，不编造咖啡因含量。 |
| 技术表达收敛 | 通过 | 用户默认视图不展示 Agent Loop、工具名等技术词。 |
| npm run build | 通过 | 构建通过，仅保留 Vite 包体积提示，不影响当前验收。 |

## 1. 产品痛点

Caffeine Coach 的核心不是让用户和一个聊天机器人聊天，而是帮助用户更快回答几个具体问题：

- 今天还能不能喝咖啡；
- 如果现在喝这杯，睡前会不会残留太多；
- 这杯饮品大概有多少咖啡因；
- 为什么系统不建议继续喝；
- 我的作息、敏感度和近期反馈如何影响建议。

因此，本项目没有把 Agent Loop 做成独立聊天页的核心卖点，而是把它接入到用户已有决策链路中：

- 首页用于回答“今天还能不能喝”；
- 喝前模拟用于回答“这杯现在喝是否合适”；
- 推荐理由用于回答“为什么这样建议”；
- 饮品 no_match 用于避免编造咖啡因含量。

这样做的好处是：用户看到的是建议卡、风险提示和可解释依据，而不是一段需要自己理解的聊天对话。

## 2. 改造前后对比

### 改造前：规则型追踪器

改造前的产品主要是一个规则驱动的轻健康记录工具：

- 记录饮品和咖啡因 mg；
- 用半衰期公式计算当前残留和睡前残留；
- 根据睡前残留判断低 / 中 / 高风险；
- 根据固定规则生成首页结论和喝前模拟结果。

优点是稳定、可控、无需后端。缺点是产品表达偏“计算器”，用户仍需要理解曲线、阈值和规则之间的关系。

### 改造后：Agent 调度型建议机制

改造后保留原有规则和计算逻辑，不重写业务系统，而是在其上方增加一层轻量 Agent Loop：

1. 识别用户意图；
2. 选择合适工具；
3. 调用工具读取记录、画像、饮品库和计算结果；
4. 汇总为普通用户能理解的结论、原因、依据和建议。

Agent Loop 不替代规则系统，而是把多个规则工具组织成一个更像“建议引擎”的产品能力。

## 3. Agent Loop 接入位置

### 首页：今日饮用建议

位置：今日页主推荐卡下方。

触发方式：

- 页面加载后；
- 饮品记录变化后；
- 设置或反馈变化后；
- 用户点击“为什么这样建议？”后刷新解释。

用户默认看到：

- 今日饮用建议；
- 今日已摄入；
- 睡前预计残留；
- 睡眠风险；
- 建议行动。

默认不展示技术细节。

### 喝前模拟

位置：喝前模拟弹窗内。

触发方式：

- 用户选择或修改模拟饮品后；
- 系统用饮品名称进入 `simulate_drink` 意图；
- 先匹配饮品库，再结合今日摄入和用户画像模拟风险。

用户默认看到：

- 模拟结论；
- 这杯饮品咖啡因；
- 喝完后今日累计；
- 睡前预计残留；
- 风险等级；
- 建议行动。

### 为什么这样建议

位置：首页建议卡和喝前模拟卡的折叠区。

展开后展示：

- 原因；
- 依据数据；
- 更深层的开发调试工具轨迹。

工具轨迹默认隐藏，标题为“开发调试：查看工具调用轨迹”。

## 4. Tool Use 工具清单

| 工具名 | 输入 | 输出 | 调用场景 |
| --- | --- | --- | --- |
| `getTodayIntakeRecords` | `date?: string` | 当日记录列表、今日总摄入量 | 首页今日建议、今日摄入摘要、解释风险 |
| `getUserProfile` | 无 | 作息时间、目标、敏感度、习惯、反馈、目标摄入量 | 首页今日建议、喝前模拟、解释风险 |
| `searchDrinkLibrary` | `drinkName: string` | `exact_match / fuzzy_match / no_match`、候选饮品、分数、匹配原因 | 记录饮品、喝前模拟、OCR 识别确认 |
| `calculateCurrentCaffeineStatus` | `records?`、`profile?`、`currentTime?` | 当前残留、睡前预计残留、睡眠风险、半衰期、睡前目标 | 首页今日建议、解释风险 |
| `simulateDrinkBeforeAdding` | `drink`、`time?`、`profile?` | 喝完后今日累计、睡前残留、风险等级、饮用建议 | 喝前模拟 |
| `generateRecommendationSummary` | `records?`、`profile?`、`calculation?` | 今日推荐摘要、风险原因、行动建议、可摄入量 | 首页今日建议、解释风险 |

所有工具统一返回：

```ts
{
  success: boolean;
  data?: unknown;
  error?: string;
  message?: string;
}
```

## 5. 核心链路

### 5.1 首页今日饮用建议链路

内部问题：

```text
今晚还能喝咖啡吗？
```

执行链路：

1. `routeIntent` 识别为 `can_i_drink`；
2. 调用 `getTodayIntakeRecords` 获取今日摄入；
3. 调用 `getUserProfile` 获取作息、目标和反馈；
4. 调用 `calculateCurrentCaffeineStatus` 计算当前残留和睡前残留；
5. 调用 `generateRecommendationSummary` 生成摘要；
6. `responseGenerator` 输出结论、原因、依据数据和建议行动；
7. 首页只默认展示轻量结果，详细依据折叠。

验收标准：

- 用户默认不看到工具名；
- 用户能快速看到今天还能不能喝；
- 展开后能看到为什么这样建议；
- 开发调试层可以看到工具调用轨迹。

### 5.2 喝前模拟链路

用户动作：

```text
选择一杯拿铁，点击喝前模拟
```

内部问题：

```text
如果我现在喝一杯拿铁会怎样？
```

执行链路：

1. `routeIntent` 识别为 `simulate_drink`；
2. `extractDrinkName` 提取饮品名；
3. 调用 `searchDrinkLibrary` 匹配饮品；
4. 若 `exact_match`，调用 `getUserProfile`；
5. 调用 `simulateDrinkBeforeAdding`；
6. 输出睡前残留、风险等级和建议行动；
7. 原因和依据折叠展示。

验收标准：

- 低 / 中 / 高风险使用不同颜色底；
- 中高风险不使用绿色背景；
- 不出现“Agent Loop”等技术文案；
- 如果饮品无法匹配，不继续模拟。

### 5.3 饮品 no_match 兜底链路

用户输入：

```text
我刚喝了一杯 XYZ987
```

执行链路：

1. `routeIntent` 识别为 `record_drink`；
2. `extractDrinkName` 提取 `XYZ987`；
3. 调用 `searchDrinkLibrary`；
4. 因无名称、别名、品牌、OCR 关键词真实命中，返回 `no_match`；
5. 候选为空；
6. 系统提示用户手动输入咖啡因含量或补充品牌 / 杯型。

返回示例：

```ts
{
  success: true,
  data: {
    matchStatus: "no_match",
    candidates: [],
    message: "暂未找到该饮品的咖啡因含量，可以手动输入或选择相近饮品。"
  }
}
```

验收标准：

- 不编造饮品；
- 不编造咖啡因 mg；
- 允许用户补充信息继续流程。

## 8. 当前版本边界

当前版本已经完成产品化接入，但仍有清晰边界：

- 当前是 mock Agent Loop，用规则路由模拟意图识别和工具调度；
- 未接真实 LLM tool calling；
- 未自动写入记录，记录类建议仍需要用户确认；
- 用户侧默认不展示技术词；
- 不做医疗诊断，只提供生活方式管理建议；
- 严重不适场景会建议停止继续摄入并咨询专业人士；
- 数据仍保存在 localStorage，没有后端数据库和登录系统。

## 9. 阶段结论

本阶段已完成 s01 Agent Loop 和 s02 Tool Use 的产品化落地。

阶段性结论：

- Agent Loop 已作为底层建议机制接入首页和喝前模拟；
- Tool Use 已覆盖记录读取、用户画像、饮品搜索、残留计算、喝前模拟和推荐摘要；
- 首页不再只是规则型追踪器，而是能给出可解释的今日饮用建议；
- 喝前模拟不只展示风险数值，也能说明为什么建议半杯、低因或暂停；
- no_match 兜底已避免未知饮品被编造为可信数据；
- 阶段 3 已完成。

下一步进入：

- Permission：让写入记录等动作进入用户确认流程；
- Hooks：把新增记录、删除记录、保存反馈等事件接入统一刷新机制；
- 记忆系统：沉淀常喝饮品、no_match 历史和用户偏好；
- Skill 文档：形成可复用的咖啡因建议生成规范；
- 评估体系：持续记录 Bad Case，验证建议稳定性和可解释性；
- 真实 LLM tool calling：在保留当前工具层的基础上替换 mock 调度。

## 11. 后续优化

### Permission

当前工具以只读建议为主。后续如果支持“帮我记录这杯”，需要增加明确确认：

- 用户确认饮品；
- 用户确认咖啡因含量；
- 用户确认写入记录；
- 写入后可撤销。

### Hooks

可在关键状态变化后触发建议刷新：

- 新增记录后；
- 删除记录后；
- 修改设置后；
- 保存反馈后；
- OCR 识别确认后。

当前首页已通过状态变化触发建议更新，后续可把刷新策略抽成统一 hook。

### 记忆系统

当前使用 localStorage 保存记录、设置和反馈。后续可增加轻量记忆：

- 用户常问问题；
- 最近 no_match 饮品；
- 常喝品牌和杯型；
- 用户偏好的解释深度；
- 用户对建议是否采纳。

### Skill 文档

可将本项目沉淀为产品级 Skill：

- 如何判断咖啡因风险；
- 如何处理 no_match；
- 如何生成克制健康建议；
- 如何避免医疗诊断表达；
- 如何解释工具调用结果。

### 真实 LLM tool calling

当前是 mock Agent Loop，后续可替换为真实 LLM tool calling：

1. 保留 `TOOL_DEFINITIONS`；
2. 保留 `TOOL_HANDLERS`；
3. 由 LLM 根据用户问题选择工具；
4. 工具结果回传 LLM 生成回复；
5. 对写入类动作增加 Permission；
6. 对健康风险表达增加安全约束。

接入真实 LLM 时，仍建议保持当前产品原则：Agent 是底层调度能力，前台展示建议卡和依据，不把产品变成聊天页。
